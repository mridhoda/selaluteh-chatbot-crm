# Validation Rules

## Purpose

Defines general backend validation rules.

## General Principles

- Validate at API boundary.
- Validate again in service layer for critical operations.
- Never trust frontend, Telegram callback, AI output, or payment payload without verification.
- Return consistent error codes.

## ID Validation

All IDs must be valid UUID after migration to Postgres.

Mongo ObjectId should only appear in migration maps, not runtime API.

## Workspace Validation

Any request touching tenant data must validate:

```txt
resource.workspace_id == current_user.workspace_id
```

For webhooks, workspace comes from resolved platform/payment.

## Product Validation

Before adding to cart:

- product exists
- product belongs to workspace
- product status is active
- variant exists if required
- variant belongs to product/workspace
- quantity valid
- stock valid if tracked

## Cart Validation

Before checkout:

- cart exists
- cart is active
- cart belongs to contact/workspace
- cart has at least one item
- all items available
- prices recalculated

## Order Validation

Before order update:

- order belongs to workspace
- status transition is allowed
- user has permission
- paid order cancellation requires reason/review

## Payment Validation

Before payment update:

- provider signature valid if webhook
- amount matches
- currency matches
- provider reference maps to payment/order
- transition allowed

## File Validation

Before file save:

- mime type allowed
- size within limit
- filename sanitized
- path generated by server
- workspace_id set

## AI Action Validation

Before executing AI action:

- action type allowed
- referenced entities exist
- workspace matches
- action does not violate payment/order/security rules
- user confirmation exists if required

## Error Codes

Recommended mapping:

| Case | Code |
|---|---|
| Invalid input | `400 Bad Request` |
| Auth required | `401 Unauthorized` |
| Permission denied | `403 Forbidden` |
| Not found in workspace | `404 Not Found` |
| Invalid status transition | `409 Conflict` |
| Rate/quota limit | `429 Too Many Requests` |
| Provider failure | `502 Bad Gateway` |
| Unexpected | `500 Internal Server Error` |

## Redaction Rule

Validation errors must not leak:

- provider secret
- DB connection string
- stack trace
- service role key
- raw token
