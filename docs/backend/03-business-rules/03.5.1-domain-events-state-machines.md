# Phase 3.5.1 - Domain Events & State Machines

## Objective

Dokumen ini mendefinisikan event bisnis, lifecycle, state machine, allowed transitions, retry, idempotency, compensation, dan kontrak payload yang menjadi penghubung antara Domain Layer, Service Layer, API, Background Jobs, Payment Provider Integration, Notification, Audit, dan Analytics.

Tujuannya adalah memastikan seluruh perubahan status di QR Store bersifat deterministik, dapat diaudit, idempotent, dan mudah dikembangkan.

## Relationship To Phase 3.5

Phase 3.5 mendefinisikan QR business domain dan domain invariants.

Phase 3.5.1 mendefinisikan bagaimana domain tersebut bergerak melalui event dan state transition.

Semua event dan transition di dokumen ini wajib mematuhi invariant pada `03.5-qr-domain-architecture.md`.

## 1. Event-Driven Domain Overview

Domain event adalah fakta bisnis yang sudah terjadi. Event bukan command, bukan request, dan bukan intent.

Canonical flow:

```txt
Customer scans QR
  -> QRCodeScanned
  -> QRSessionCreated
  -> CartCreated
  -> CartItemAdded
  -> CheckoutStarted
  -> CheckoutValidated
  -> PaymentCreated
  -> PaymentWaitingProvider
  -> PaymentSucceeded
  -> OrderPaid
  -> OrderAccepted
  -> OrderPreparing
  -> OrderReady
  -> OrderCompleted
```

Rules:

- Event name uses past tense or completed business fact.
- Event is immutable after published.
- Event payload must include workspace scope.
- Event payload must include a unique event ID.
- Event consumer must be idempotent.
- Event publication must happen after the state change is committed, or through an outbox pattern in the same transaction.

## 2. Event Taxonomy

Event groups:

- QR events.
- QR session events.
- Cart events.
- Checkout events.
- Payment events.
- Order events.
- Fulfillment events.
- Notification events.
- Audit events.
- Analytics events.
- Compensation events.

Event severity:

| Severity | Meaning | Examples |
|---|---|---|
| Critical | Affects money, order, payment, or security | `PaymentSucceeded`, `OrderPaid`, `OrderCancelled` |
| Important | Affects customer flow or fulfillment | `CheckoutRejected`, `OrderReady`, `QRSessionExpired` |
| Informational | Useful for analytics or notification | `QRCodeScanned`, `NotificationSent` |

Event delivery class:

| Class | Requirement | Examples |
|---|---|---|
| Exactly-once effect | Consumer may receive duplicates, but business effect must happen once | `PaymentSucceeded`, `OrderPaid` |
| At-least-once delivery | Retry until successful or dead-lettered | `NotificationRequested`, `AuditLogCreated` |
| Best-effort | Failure must not block checkout/order | Analytics events |

## 3. Shared Event Envelope

Every domain event should use a consistent envelope.

```json
{
  "eventId": "evt_01H...",
  "eventName": "PaymentSucceeded",
  "eventVersion": 1,
  "occurredAt": "2026-07-07T00:00:00.000Z",
  "workspaceId": "wrk_123",
  "correlationId": "corr_123",
  "causationId": "evt_or_request_123",
  "actor": {
    "type": "customer|admin|system|provider|job|ai",
    "id": "optional_actor_id"
  },
  "source": "PaymentProviderService",
  "payload": {}
}
```

Required envelope fields:

- `eventId`: globally unique event identifier.
- `eventName`: canonical event name.
- `eventVersion`: integer version for payload evolution.
- `occurredAt`: timestamp when the business fact occurred.
- `workspaceId`: tenant boundary.
- `correlationId`: groups events in one business flow.
- `causationId`: request ID or previous event ID that caused this event.
- `actor`: customer, admin, system, provider, job, or AI.
- `source`: service that published the event.
- `payload`: event-specific data.

|---|---|---|---|
| `QRCodeCreated` | Admin creates QR | `QrService` | `AuditService` |
| `QRCodeActivated` | QR becomes active | `QrService` | `AuditService`, `AnalyticsService` |
| `QRCodeDisabled` | QR temporarily disabled | `QrService` | `AuditService` |
| `QRCodeRevoked` | QR permanently revoked | `QrService` | `AuditService`, `AnalyticsService` |
| `QRCodeScanned` | Customer scans valid QR | `QrService` | `QrSessionService`, `AnalyticsService` |
| `QRCodeExpired` | QR passes expiry or job marks expired | `QrExpiryJob` | `AuditService`, `AnalyticsService` |

Example payload:

```json
{
  "qrId": "qr_123",
  "qrScope": "universal|outlet|location",
  "storefrontId": "stf_123",
  "outletId": "out_123",
  "locationId": "loc_123",
  "scanContext": {
    "ipHash": "optional_hash",
    "userAgentHash": "optional_hash",
    "referrer": "optional_referrer"
  }

```

Rules:

- `QRCodeScanned` must not be emitted for revoked or expired QR.
- `QRCodeRevoked` is terminal unless a future admin recovery policy explicitly allows regeneration as a new QR identity.
- QR events never change payment or order state directly.

```json
{
  "sessionId": "qrs_123",
```

- One QR session may produce at most one successful checkout.
- Expired session cannot transition back to active.
- Recovery must preserve original workspace, QR, outlet, and location context.

|---|---|---|---|
| `PaymentCreated` | Backend creates payment record | `PaymentService` | `AuditService` |

| `PaymentExpired` | Payment exceeds expiry | `PaymentExpiryJob` | `OrderService`, `NotificationService`, `AuditService` |
| `PaymentCancelled` | Customer/admin cancels before paid | `PaymentService` | `OrderService`, `AuditService` |
| `PaymentRefundRequested` | Authorized refund requested | `PaymentService` | `AuditService` |
| `PaymentRefunded` | Provider confirms refund | `PaymentProviderService` | `OrderService`, `NotificationService`, `AuditService` |

Example payload:

```json
{
  "paymentId": "pay_123",
  "orderId": "ord_123",
  "provider": "xendit|midtrans|manual",
  "providerReference": "provider_ref_123",
  "environment": "sandbox|production",

  "currency": "IDR",
  "previousStatus": "pending",
  "currentStatus": "paid"

```

Rules:

- `PaymentSucceeded` may only come from verified webhook or active backend verification.
- Duplicate provider callbacks must not create duplicate `OrderPaid` effects.
- Payment failure, expiry, and cancellation must not erase payment history.

## 9. Order Events

| Event | Published When | Publisher | Primary Subscribers |
|---|---|---|---|
| `OrderCreated` | Order record created | `CheckoutService` | `AuditService`, `AnalyticsService` |
| `OrderPendingPayment` | Order awaits payment | `OrderService` | `NotificationService` |
| `OrderPaid` | Payment success marks order paid | `OrderService` | `FulfillmentService`, `NotificationService`, `AuditService` |
| `OrderAccepted` | Outlet accepts order | `OrderService` | `FulfillmentService`, `NotificationService`, `AuditService` |
| `OrderRejected` | Outlet rejects order where policy allows | `OrderService` | `PaymentService`, `NotificationService`, `AuditService` |
| `OrderPreparing` | Preparation starts | `OrderService` | `NotificationService`, `AuditService` |
| `OrderReady` | Order ready for pickup/serve | `OrderService` | `NotificationService`, `AuditService` |
| `OrderCompleted` | Order fulfilled | `OrderService` | `NotificationService`, `AnalyticsService`, `AuditService` |
| `OrderCancelled` | Order cancelled | `OrderService` | `PaymentService`, `NotificationService`, `AuditService` |

Rules:

- Order state cannot skip required transitions.
- `OrderPaid` must be caused by `PaymentSucceeded`.
- `OrderCompleted` must only occur after fulfillment reaches a terminal successful state.
- Historical order snapshots must remain immutable across all order events.

## 10. Fulfillment Events

| Event | Published When | Publisher | Primary Subscribers |
|---|---|---|---|
| `FulfillmentWaiting` | Order is paid and waiting outlet action | `FulfillmentService` | `NotificationService` |
| `PickupStarted` | Pickup preparation begins | `FulfillmentService` | `NotificationService`, `AuditService` |
| `PickupReady` | Pickup is ready | `FulfillmentService` | `NotificationService`, `AuditService` |
| `PickupCompleted` | Pickup completed | `FulfillmentService` | `OrderService`, `AnalyticsService` |
| `DineInPreparing` | Dine-in preparation begins | `FulfillmentService` | `NotificationService`, `AuditService` |
| `DineInServed` | Dine-in order served | `FulfillmentService` | `OrderService`, `AnalyticsService` |
| `DeliveryAssigned` | Future delivery assigned | `FulfillmentService` | `NotificationService` |
| `DeliveryCompleted` | Future delivery completed | `FulfillmentService` | `OrderService`, `AnalyticsService` |

Rules:

- Fulfillment event must reference one order and one outlet.
- Fulfillment cannot start before order is paid unless a future explicit pay-later policy exists.
- Delivery events are future-ready and must not be required by MVP pickup/dine-in flows.

| Event | Published When | Publisher | Primary Subscribers |
|---|---|---|---|
| `NotificationRequested` | Domain flow requests notification | `NotificationService` | Notification worker |
| `PaymentSuccessNotificationRequested` | Customer should be notified payment succeeded | `NotificationService` | Notification worker |
| `OrderReadyNotificationRequested` | Customer should be notified order ready | `NotificationService` | Notification worker |

## 13. Order State Machine

Canonical order states:

```txt
draft
  -> pending_payment
  -> paid
  -> accepted
  -> preparing
  -> ready
  -> completed
```

Failure or alternate branches:

```txt
pending_payment -> payment_failed
pending_payment -> payment_expired
pending_payment -> cancelled
paid            -> cancelled          (admin/outlet policy only)
paid            -> rejected           (outlet policy only)
accepted        -> cancelled          (admin policy only)
preparing       -> cancelled          (admin exceptional policy only)
```

Rules:

- `paid` is reached only from verified payment success.
- `accepted`, `preparing`, `ready`, and `completed` require `payment_status=paid`.
- `completed`, `cancelled`, `rejected`, `payment_failed`, and `payment_expired` are terminal for normal customer flow.

## 14. Payment State Machine

Canonical payment states:

```txt
created
  -> pending
  -> waiting_provider
  -> paid
```

Failure or alternate branches:

```txt
pending          -> expired
pending          -> failed
pending          -> cancelled
waiting_provider -> expired
waiting_provider -> failed
waiting_provider -> cancelled
paid             -> refund_requested
refund_requested -> refunded
```

Rules:

- `paid` is terminal for payment collection, except refund flow.
- `expired`, `failed`, `cancelled`, and `refunded` are terminal.
- `paid` must not be set by frontend or unverified callback.

## 15. QR State Machine

Canonical QR states:

```txt
created
  -> active
  -> disabled
  -> active
```

Terminal branches:

```txt
active   -> revoked
active   -> expired
disabled -> revoked
disabled -> expired
```

Rules:

- Only `active` QR can create new sessions.
- `revoked` is terminal.
- `expired` is terminal unless future policy explicitly creates a new QR identity.
- `disabled` is reversible by authorized admin.

## 16. QR Session State Machine

Canonical QR session states:

```txt
created
  -> browsing
  -> cart_active
  -> checkout
  -> completed
```

Failure or alternate branches:

```txt
created     -> expired
browsing    -> expired
cart_active -> expired
checkout    -> expired
checkout    -> closed
```

Rules:

- `completed`, `expired`, and `closed` are terminal.
- A completed session cannot create another checkout.
- Session state must preserve original QR, outlet, and location context.

## 17. Checkout State Machine

Canonical checkout states:

```txt
created
  -> validating
  -> payment_creating
  -> order_creating
  -> confirmed
```

Failure or alternate branches:

```txt
validating       -> rejected
payment_creating -> failed
order_creating   -> failed
created          -> expired
validating       -> expired
```

Rules:

- `confirmed`, `rejected`, `failed`, and `expired` are terminal.
- `confirmed` means order and payment have been created atomically.
- Failed checkout must not leave partial order/payment artifacts outside compensation handling.

## 18. Fulfillment State Machine

Pickup:

```txt
waiting
  -> accepted
  -> preparing
  -> ready
  -> picked_up
  -> completed
```

Dine in:

```txt
waiting
  -> accepted
  -> preparing
  -> served
  -> completed
```

Failure or alternate branches:

```txt
waiting   -> rejected
accepted  -> cancelled
preparing -> cancelled   (admin exceptional policy only)
```

Rules:

- Fulfillment starts only after payment is paid.
- `completed`, `cancelled`, and `rejected` are terminal.
- Fulfillment type must not change after outlet accepts order unless through audited admin override.

## 20. Compensation Rules

Compensation is required when a process partially succeeds across external boundaries.

High-risk cases:

| Failure Case | Required Response |
|---|---|
| Provider payment succeeds but webhook processing fails | Retry webhook processing idempotently |
| `PaymentSucceeded` processed but `OrderPaid` update fails | Retry order transition, then manual review |
| Checkout creates payment but order write fails | Rollback transaction; if provider object exists, mark checkout failed and cancel/expire provider payment where possible |
| Notification fails after order ready | Retry notification; do not roll back order |
| Audit write fails for critical event | Persist to fallback/dead-letter and alert operations |

Compensation flow:

```txt
Detect failure
  -> Retry idempotently
  -> Dead-letter if retry exhausted
  -> Manual review if money/order/security is affected
  -> Publish compensation event
```

Compensation events:

- `CompensationRequested`.
- `CompensationRetryScheduled`.
- `CompensationManualReviewRequired`.
- `CompensationCompleted`.
- `CompensationFailed`.

## 21. Retry Strategy

| Process | Retry Policy | Notes |
|---|---|---|
| Payment webhook processing | 5 attempts with exponential backoff | Must be idempotent by provider reference/event ID |
| Payment provider verification | 3 attempts with backoff | Stop if provider returns terminal status |
| Notification delivery | 3 attempts with backoff | Failure must not roll back business state |
| Audit log write | 5 attempts, then fallback/dead-letter | Critical for money/order/security |
| Analytics event | 3 attempts or best-effort | Must not block customer flow |
| Expiry jobs | Retry on next scheduler tick | Must be idempotent by entity ID and target state |

Retry rules:

- Retried operation must use same idempotency key.
- Retry must not create duplicate order, payment, audit log, or notification.
- Terminal states must stop retry unless compensation policy says otherwise.

## 22. Idempotency Rules

Every important event must be idempotent.

Required idempotency keys:

| Flow | Idempotency Key |
|---|---|
| QR scan session creation | `qrId + customerContextId + activeSessionWindow` |
| Checkout confirmation | `sessionId + checkoutId` |
| Payment provider callback | `provider + providerReference + providerEventId` |
| Order paid transition | `orderId + PaymentSucceeded.eventId` |
| Notification | `notificationType + entityId + targetRecipient` |
| Audit log | `eventId + action + entityId` |

Example:

```txt
PaymentSucceeded webhook arrives 10 times
  -> Payment state becomes paid once
  -> OrderPaid is emitted once
  -> Customer payment success notification is requested once
  -> Audit log is written once
```

## 23. Event Ordering

Required ordering examples:

Correct:

```txt
PaymentCreated
  -> PaymentWaitingProvider
  -> PaymentSucceeded
  -> OrderPaid
```

Invalid:

```txt
OrderPaid
  -> PaymentCreated
```

Correct:

```txt
QRCodeScanned
  -> QRSessionCreated
  -> CartCreated
  -> CheckoutStarted
```

Invalid:

```txt
CheckoutConfirmed
  -> QRSessionCreated
```

Ordering rules:

- Subscriber must not observe impossible states.
- Event must be published after the state change it represents.
- If events cross process boundaries, use outbox ordering by aggregate ID.
- Within one aggregate, preserve event order.
- Across unrelated aggregates, strict global ordering is not required.

## 24. Future Event Bus Compatibility

MVP may run as a monolith, but event design must stay transport-agnostic.

Future-compatible transports:

- RabbitMQ.
- Kafka.
- Redis Streams.
- Azure Service Bus.
- Google Pub/Sub.
- AWS SNS/SQS.

Compatibility rules:

- Domain event must not depend on HTTP request object, ORM model instance, or provider SDK object.
- Payload must be serializable JSON.
- Event version must support schema evolution.
- Consumer must tolerate unknown fields.
- Publisher must not remove existing required fields without version bump.
- Outbox table or durable event log is recommended before introducing external event bus.
