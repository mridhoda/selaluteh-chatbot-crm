# Phase 3.7 - REST API Architecture

## Objective

Dokumen ini mendefinisikan arsitektur REST API untuk SELKOP Online Store dan QR Store agar komunikasi antara frontend, admin dashboard, payment provider, worker/internal tooling, dan backend berjalan konsisten, aman, versioned, dan mudah dites.

Phase ini menjembatani:

```txt
Frontend / Admin Dashboard / Payment Provider
  -> REST API Layer
  -> Application Service Layer
  -> Repository Layer
  -> Database
```

## Implementation Status

Sebagian besar fondasi runtime untuk phase ini sudah ada:

- Public customer routes berjalan melalui `server/src/routes/public-store.js` dan dimount di `/api/public` serta `/api/v1/public`.
- Public storefront route mendukung `/api/v1/public/stores/:storefrontSlug` dan alias `/api/v1/public/storefronts/:storefrontSlug`.
- Public QR, cart validation, checkout, payment status, dan public order status sudah berada di route public-store.
- Admin order aliases berjalan melalui `server/src/routes/admin-orders.js` dan dimount di `/api/v1/admin/orders`.
- Payment webhook routes berjalan melalui `server/src/routes/webhooks/*` dan dimount di `/api/v1/webhooks`.
- Public route rate limits sudah diterapkan melalui `server/src/middleware/rate-limit.js`.
- OpenAPI contract untuk phase berikutnya tersedia di `docs/api/openapi.yaml` dan dijelaskan di `03.7.1-openapi-specification.md`.

Beberapa endpoint target seperti admin QR management dan payment provider settings masih kontrak desain/target alias, bukan seluruhnya route runtime final.

## 1. API Architecture Principles

Controllers harus tipis. Tanggung jawab controller:

- Parse request.
- Validate auth context.
- Validate required route/query/body shape.
- Call application service.
- Map service result to HTTP response.
- Return status code and response.

Controller tidak boleh:

- Mengakses database langsung.
- Mengubah payment status langsung.
- Mengubah order status tanpa service/state machine.
- Mempercayai total/price dari frontend.
- Mengembalikan raw database entity tanpa public/admin serializer.
- Mencetak secret atau raw provider payload ke log.

Flow yang benar:

```txt
Route / Controller
  -> Middleware auth/rate-limit/request-id
  -> Application Service
  -> Repository
  -> Database
```

## 2. API Base URL

Base API final:

```http
/api/v1
```

Route group utama:

```txt
/api/v1/public/*
/api/v1/admin/*
/api/v1/webhooks/*
/api/v1/internal/*
```

Legacy/compatibility mounts boleh tetap hidup selama migrasi, seperti `/api/public`, `/api/payments`, `/orders`, atau `/webhook`. Endpoint marketplace baru harus memakai `/api/v1`.

## 3. API Boundary

### Public API

Public API dipakai customer untuk online store, QR store, order status page, dan payment status page. Public bukan berarti trusted.

Public API harus punya:

- Opaque public token.
- QR session token jika QR context diperlukan.
- Rate limit.
- Input validation.
- Public-safe serializer.

### Admin API

Admin API dipakai staff/admin untuk order management, product availability, QR management, dan payment provider settings.

Admin API wajib punya:

- Authentication.
- Workspace scope.
- RBAC permission.
- Outlet scope jika endpoint outlet-specific.
- Audit log untuk mutation penting.

### Webhook API

Webhook API dipakai provider eksternal, misalnya payment provider atau platform messaging.

Webhook API wajib punya:

- Provider signature/token verification.
- Payload size limit.
- Idempotency key/event key.
- Fast response.
- Service/state-machine processing.

### Internal API

Internal API disiapkan untuk worker, system integration, internal tooling, atau future microservices.

Internal API tidak boleh public tanpa trusted runtime/auth layer.

`/stores/:storefrontSlug` adalah runtime alpha route awal. `/storefronts/:storefrontSlug` adalah alias yang lebih eksplisit untuk kontrak baru.

## 5. Public Storefront API

Endpoint:

```http
GET /api/v1/public/storefronts/:storefrontSlug
GET /api/v1/public/stores/:storefrontSlug
```

Purpose:

- Load storefront.
- Load brand metadata.
- Load active/orderable outlets.
- Load categories.
- Load products.
- Load customer-safe availability.

Public response tidak boleh expose:

- Internal product cost.
- Admin notes.
- Provider config.
- Secret.
- Audit logs.
- Raw DB fields yang tidak dibutuhkan customer.

## 6. Public QR API

Endpoint:

```http
GET /api/v1/public/qr/:qrToken
```

Purpose:

- Resolve QR token.
- Validate QR status.
- Create or refresh QR session.
- Return storefront context.
- Return outlet/location resolution rule.

Response variants:

```txt
universal QR -> allow outlet selection
outlet QR    -> lock outlet
location QR  -> lock outlet and table/location
```

Rules:

- Raw QR token must not be stored as plain text.
- Expired/revoked QR must fail safely.
- QR session expired must not be usable for new checkout.

## 7. Cart Validation API

Endpoint:

```http
POST /api/v1/public/carts/validate
```

Purpose:

- Validate cart before checkout.
- Recalculate authoritative price.
- Check product availability.
- Check outlet validity.
- Check modifier validity.
- Return final payable preview.

Frontend may send:

- `product_id`.
- `quantity`.
- `modifier_option_ids`.
- `note`.
- `outlet_id`.
- `qr_session_token`.

Frontend must not control:

- `unit_price`.
- `subtotal`.
- `total`.
- `discount`.
- `payment_status`.
- `order_status`.

## 8. Checkout API

Endpoint:

```http
POST /api/v1/public/checkout
Idempotency-Key: <unique-key>
```

Purpose:

- Validate checkout input.
- Create checkout/order snapshot.
- Create order items.
- Create payment attempt/session.
- Call payment provider outside database transaction.
- Return payment URL/QRIS and order tracking token.

Rules:

- `Idempotency-Key` wajib.
- Same key + same payload returns current result.
- Same key + different payload returns `IDEMPOTENCY_CONFLICT`.
- Backend remains source of truth for totals.

## 9. Payment Status API

Endpoint:

```http
GET /api/v1/public/payments/:paymentId/status
```

Purpose:

- Frontend polling payment status.
- Show payment URL/expiry/status safely.
- Link to public order status.

Rules:

- Endpoint read-only.
- Frontend cannot mark payment paid.
- Paid status only comes from valid webhook or provider verification/reconciliation.
- No raw provider payload, workspace ID, or secret in response.

## 10. Public Order Status API

Endpoint:

```http
GET /api/v1/public/orders/:publicOrderToken
```

Allowed public fields:

- `order_number`.
- `public_order_status`.
- `payment_status` public-safe.
- `fulfillment_status` public-safe.
- Item snapshot.
- Outlet snapshot.
- Fulfillment type.
- Created/updated timestamps if needed.
- Masked customer phone.

Forbidden public fields:

- `internal_note`.
- Admin user fields.
- Audit logs.
- Raw payment provider payload.
- Full provider metadata.
- Full customer phone.

## 12. Admin Order API Rules

Admin tidak boleh melakukan generic status patch seperti:

```http
PATCH /api/v1/admin/orders/:id
{ "status": "completed" }
```

Gunakan action endpoint:

```http
POST /api/v1/admin/orders/:id/complete
```

Rules:

- Each action maps to business rule and state machine transition.
- Detail/list response includes `allowed_actions` as UI hint.
- Service transition guard remains authoritative.
- Cancel requires non-empty reason.
- Hard delete order disabled; use cancel.

## 13. Admin Product Availability API

Target endpoint:

```http
GET  /api/v1/admin/products/:productId/availability
POST /api/v1/admin/products/:productId/availability
```

Runtime alpha compatibility:

```http
GET /api/products/:productId/outlet-availability
PUT /api/products/:productId/outlet-availability
```

Rule:

- Availability selalu outlet-specific.
- Staff can mark product available/sold out per outlet.
- Product price/availability response must not expose internal cost.

## 14. Admin QR API

Target endpoints:

```http
GET  /api/v1/admin/qr-codes
POST /api/v1/admin/qr-codes
GET  /api/v1/admin/qr-codes/:qrCodeId
POST /api/v1/admin/qr-codes/:qrCodeId/activate
POST /api/v1/admin/qr-codes/:qrCodeId/disable
POST /api/v1/admin/qr-codes/:qrCodeId/revoke
```

Rules:

- QR is not hard deleted.
- QR that has been used should only move to disabled, revoked, or expired.
- QR token must remain opaque and unguessable.
- QR management requires `qr.manage` or equivalent permission.

## 15. Payment Provider Settings API

Target endpoints:

```http
GET  /api/v1/admin/settings/payment-providers
POST /api/v1/admin/settings/payment-providers/:providerCode/activate
POST /api/v1/admin/settings/payment-providers/:providerCode/configure
```

Rules:

- Required permission: `settings.payment.manage`.
- Must create audit log.
- Only one active provider per workspace per mode.
- Secrets are write-only and never returned.
- Existing payments keep their original provider.

## 16. Webhook API

Endpoint:

```http
POST /api/v1/webhooks/payments/:providerCode
```

Purpose:

- Receive payment event.
- Verify provider signature/token.
- Store webhook event or payload hash.
- Process idempotently.
- Update payment status through service/state machine.
- Publish domain event or enqueue background job.

Ideal flow:

```txt
Receive webhook
  -> verify signature
  -> store event
  -> enqueue processing job
  -> return 200/202 quickly
```

Duplicate webhook returns a safe no-op success and must not duplicate payment event, order transition, audit log, or notification.

## 17. Authentication Strategy

Public API:

- No login required by default.
- Requires public token/QR token/order token where relevant.
- Always rate-limited.
- Always input-validated.

Admin API:

- Requires bearer JWT.
- Attaches user and workspace context.
- Checks permission and outlet scope.

Webhook API:

- Requires provider signature/token.
- Validates provider reference, amount, currency, and payload hash before mutation.

## 19. Standard Response Shape

Target success shape:

```json
{
  "success": true,
  "data": {},
  "meta": {
    "request_id": "req_01HXABC123"
  }
}
```

Target error shape:

```json
{
  "success": false,
  "error": {
    "code": "PRODUCT_UNAVAILABLE",
    "message": "Produk tidak tersedia di outlet ini.",
    "details": {}
  },
  "meta": {
    "request_id": "req_01HXABC123"
  }
}
```

Runtime existing may return lighter envelopes such as `{ data }` or `{ order }`. New endpoints should converge to a stable documented shape and OpenAPI must reflect current behavior during migration.

Stable error codes:

- `VALIDATION_ERROR`.
- `AUTHENTICATION_REQUIRED`.
- `FORBIDDEN`.
- `NOT_FOUND`.
- `CONFLICT`.
- `RATE_LIMITED`.
- `PAYMENT_PROVIDER_ERROR`.
- `PAYMENT_WEBHOOK_INVALID`.
- `QR_EXPIRED`.
- `QR_REVOKED`.
- `QR_OUTLET_MISMATCH`.
- `QR_LOCATION_MISMATCH`.
- `PRODUCT_UNAVAILABLE`.
- `CHECKOUT_EXPIRED`.
- `IDEMPOTENCY_CONFLICT`.
- `ORDER_INVALID_TRANSITION`.

Frontend must rely on `error.code`, not `error.message`.

## 21. HTTP Status Convention

| Case | HTTP Status |
|---|---:|
| Success | `200` |
| Created | `201` |
| Accepted async processing | `202` |
| Validation error | `400` |
| Unauthorized | `401` |
| Forbidden | `403` |
| Not found | `404` |
| Conflict / invalid state | `409` |
| Rate limited | `429` |
| Provider error | `502` |
| Unexpected error | `500` |

## 22. Pagination, Filtering, Sorting

For list APIs:

```http
GET /api/v1/admin/orders?page=1&page_size=20
```

Common filters:

- `outlet_id`.
- `channel`.
- `payment_status`.
- `fulfillment_status`.
- `date_from`.
- `date_to`.
- `search`.

Sorting:

```http
?sort=created_at.desc
```

Pagination response:

```json
{
  "data": [],
  "pagination": {
    "page": 1,
    "page_size": 20,
    "total": 120,
    "total_pages": 6
  }
}
```

Runtime alpha may use `limit` as alias for `page_size`.

## 23. Idempotency Rules

Endpoints that must be idempotent:

- `POST /api/v1/public/checkout`.
- `POST /api/v1/webhooks/payments/:providerCode`.
- `POST /api/v1/admin/orders/:id/accept`.
- `POST /api/v1/admin/orders/:id/prepare`.
- `POST /api/v1/admin/orders/:id/ready`.
- `POST /api/v1/admin/orders/:id/complete`.
- `POST /api/v1/admin/orders/:id/cancel`.

Checkout requires `Idempotency-Key` header. Admin actions are idempotent by state-machine guard and current status check.

## 24. Rate Limiting

Minimal alpha policy:

| Endpoint | Limit |
|---|---:|
| `GET /api/v1/public/qr/:token` | `60/min/IP` |
| `POST /api/v1/public/carts/validate` | `30/min/IP/session` |
| `POST /api/v1/public/checkout` | `10/min/IP/session` |
| `GET /api/v1/public/payments/:id/status` | `30/min/order` |
| `GET /api/v1/public/orders/:token` | `60/min/token` |
| Admin login | `5/min/IP/email` |
| Admin order action | `60/min/user` |

Webhook should not rely on public rate limit alone. Webhook security depends on signature verification, payload size limit, idempotency, and optional provider allowlist.

## 26. API Versioning Strategy

Current version:

```txt
/api/v1
```

Rules:

- Breaking changes go to `/api/v2`.
- Non-breaking additive fields can remain in `/api/v1`.
- Existing fields should not be removed without deprecation.
- Deprecated endpoint should be marked in OpenAPI.
- OpenAPI version should track alpha contract changes.

## 28. API Testing Strategy

Minimum test coverage:

Public QR:

- Universal QR response.
- Outlet QR response.
- Location QR response.
- Expired QR.
- Revoked QR.

Checkout:

- Valid checkout.
- Invalid product.
- Outlet mismatch.
- Location mismatch.
- Duplicate idempotency key.
- Payment provider error.

Payment:

- Pending status.
- Paid status from webhook/reconciliation.
- Failed status.
- Expired status.

Admin:

- Accept paid order.
- Reject unpaid order transition.
- Prepare order.
- Ready order.
- Complete order.
- Cancel with reason.

Webhook:

- Valid signature.
- Invalid signature.
- Duplicate webhook.
- Amount mismatch.
- Unknown provider reference.

OpenAPI contract tests are specified in `03.7.1-openapi-specification.md`.
