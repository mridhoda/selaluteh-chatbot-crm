# Phase 3.7.1 - OpenAPI Specification

## Objective

Dokumen ini mendefinisikan kontrak formal REST API SELKOP dalam format OpenAPI. Kontrak ini menjadi sumber bersama untuk frontend, admin dashboard, backend controller, payment webhook, QA, dan AI coding agent.

Deliverable teknis:

- `docs/api/openapi.yaml`

OpenAPI menjawab bentuk request, response, schema, error, auth, pagination, dan contoh payload untuk endpoint alpha.

## Implementation Status

Phase ini sudah diterapkan sebagai dokumentasi kontrak dan file machine-readable:

- `docs/api/openapi.yaml` memakai OpenAPI `3.1.0`.
- Endpoint public alpha terdaftar di bawah `/api/v1/public`.
- Endpoint admin orders terdaftar di bawah `/api/v1/admin/orders`.
- Endpoint product availability terdaftar memakai route runtime saat ini di `/api/products/{productId}/outlet-availability` dan target alias `/api/v1/admin/products/{productId}/availability`.
- Endpoint payment webhook terdaftar di `/api/v1/webhooks/payments/{providerCode}`.
- Common schemas, enums, error catalog, security schemes, examples, dan pagination contract sudah didefinisikan.

Beberapa endpoint di OpenAPI bersifat target contract untuk arsitektur `/api/v1/admin/*` dan belum semuanya memiliki route alias runtime. Endpoint yang belum runtime-ready diberi tag/description yang menyebut status target alias.

## 1. OpenAPI Purpose

OpenAPI digunakan untuk:

- API documentation.
- Mock server.
- TypeScript client generation.
- Backend route stub generation.
- Request validation.
- Response validation.
- Contract test.
- Postman collection.
- Swagger UI atau Redoc.

Kontrak OpenAPI wajib menjadi referensi ketika route baru dibuat atau field response berubah.

## 2. OpenAPI Version

Gunakan:

```yaml
openapi: 3.1.0
```

Metadata API:

```yaml
info:
  title: SELKOP Online & QR Store API
  version: 1.0.0-alpha
```

Base URL utama:

```txt
/api/v1
```

Legacy mount seperti `/api/public`, `/api/payments`, dan `/orders` masih boleh ada selama migrasi, tetapi kontrak marketplace baru harus bergerak ke `/api/v1`.

## 4. Security Schemes

Security schemes di `openapi.yaml`:

- `AdminBearerAuth`: JWT untuk admin/staff API.
- `QrSessionToken`: public QR session token di header `X-QR-Session-Token`.
- `IdempotencyKey`: stable idempotency key di header `Idempotency-Key`.
- `ProviderWebhookSignature`: provider signature di header `X-Provider-Signature`.
- `XenditCallbackToken`: Xendit callback token di header `x-callback-token`.

Rule:

- Public API tetap divalidasi dan di-rate-limit walaupun tidak memakai login.
- Admin API wajib auth, workspace scope, permission, dan outlet scope jika relevan.
- Webhook API wajib signature/token verification sebelum state mutation.

## 5. Common Response Contract

Canonical response bentuk ideal:

```json
{
  "success": true,
  "data": {},
  "meta": {
    "request_id": "req_01HXABC123",
    "timestamp": "2026-07-07T02:00:00.000Z"
  }
}
```

Runtime existing belum semua route memakai wrapper `success/data/meta`. Karena itu `openapi.yaml` mendokumentasikan bentuk response runtime-safe per endpoint sekaligus menyediakan schema `SuccessResponse`, `ErrorResponse`, dan `ResponseMeta` sebagai target standardization.

Error response:

```json
{
  "success": false,
  "error": {
    "code": "PRODUCT_UNAVAILABLE",
    "message": "Produk tidak tersedia di outlet ini.",
    "details": {}
  },
  "meta": {
    "request_id": "req_01HXABC123"
  }
}
```

Frontend harus bergantung pada `error.code`, bukan parsing `error.message`.

## 7. Public Storefront API

Endpoint:

```http
GET /api/v1/public/stores/{storefrontSlug}
```

Purpose:

- Load storefront metadata.
- Load outlets aktif/orderable.
- Load menu public.
- Load availability yang aman untuk customer.

Public response tidak boleh expose internal cost, admin note, provider config, secret, audit log, atau raw DB entity internal.

## 8. Public QR API

Endpoint:

```http
GET /api/v1/public/qr/{qrToken}
```

Purpose:

- Resolve QR token.
- Validate QR state.
- Create/refresh QR session.
- Return store context.
- Return outlet/location lock jika QR scope bukan universal.

Response memakai schema `QRResolveResponse` dengan varian universal, outlet, atau location.

Error penting:

- `QR_EXPIRED`.
- `QR_REVOKED`.
- `QR_NOT_FOUND`.
- `OUTLET_CLOSED`.
- `ORDERING_DISABLED`.

## 9. Cart Validation API

Endpoint:

```http
POST /api/v1/public/carts/validate
```

Rule:

- Frontend boleh mengirim item selection, outlet, fulfillment type, dan QR token/session.
- Frontend tidak boleh menjadi source of truth untuk final price.
- Backend recompute subtotal, tax, fee, discount, total, dan availability.

## 10. Checkout API

Endpoint:

```http
POST /api/v1/public/checkout
Idempotency-Key: <key>
```

Purpose:

- Validate cart.
- Create immutable order snapshot.
- Create payment attempt/session.
- Return payment URL/QRIS and public order token.

Rule:

- `Idempotency-Key` wajib.
- Same key and same payload returns same result.
- Same key with different payload returns `IDEMPOTENCY_CONFLICT`.
- Frontend tidak boleh mengirim final payable amount sebagai authority.

## 11. Payment Status API

Endpoint:

```http
GET /api/v1/public/payments/{paymentId}/status
```

Rule:

- Endpoint read-only.
- Tidak ada endpoint public untuk mark payment paid.
- Paid hanya dari provider webhook valid atau provider verification/reconciliation.
- Response tidak expose raw provider payload, reconciliation internals, workspace ID, atau secret.

## 12. Public Order API

Endpoint:

```http
GET /api/v1/public/orders/{publicOrderToken}
```

Allowed public fields:

- Order number.
- Public order status.
- Payment status.
- Fulfillment status.
- Outlet snapshot.
- Item snapshot.
- Total amount.
- Currency.
- Masked customer phone.

Forbidden fields:

- `internal_note`.
- Audit logs.
- Raw provider payload.
- Admin user ID.
- Full customer phone.

## 13. Admin Order API

Endpoints:

```http
GET  /api/v1/admin/orders
GET  /api/v1/admin/orders/{orderId}
POST /api/v1/admin/orders/{orderId}/accept
POST /api/v1/admin/orders/{orderId}/prepare
POST /api/v1/admin/orders/{orderId}/ready
POST /api/v1/admin/orders/{orderId}/complete
POST /api/v1/admin/orders/{orderId}/cancel
```

Rules:

- Semua endpoint memakai `AdminBearerAuth`.
- List endpoint memakai pagination.
- Detail response include `allowed_actions`.
- Admin mutation harus melewati action endpoint, bukan generic status update.
- Cancel request wajib punya `reason`.

## 14. Admin Product Availability API

Runtime route saat ini:

```http
GET /api/products/{productId}/outlet-availability
PUT /api/products/{productId}/outlet-availability
```

Target `/api/v1` alias:

```http
GET  /api/v1/admin/products/{productId}/availability
POST /api/v1/admin/products/{productId}/availability
```

Rules:

- Availability outlet-specific.
- Admin/staff harus punya permission product write/read.
- Sold-out window boleh memakai nullable date-time.

## 15. Admin QR API

Target endpoints:

```http
GET  /api/v1/admin/qr-codes
POST /api/v1/admin/qr-codes
GET  /api/v1/admin/qr-codes/{qrCodeId}
POST /api/v1/admin/qr-codes/{qrCodeId}/activate
POST /api/v1/admin/qr-codes/{qrCodeId}/disable
POST /api/v1/admin/qr-codes/{qrCodeId}/revoke
```

Validation rule:

- `scope = universal`: `outlet_id` null and `qr_location_id` null.
- `scope = outlet`: `outlet_id` required.
- `scope = location`: `outlet_id` and `qr_location_id` required.

QR code tidak boleh hard delete sebagai normal admin action.

## 16. Payment Provider Settings API

Target endpoints:

```http
GET  /api/v1/admin/settings/payment-providers
POST /api/v1/admin/settings/payment-providers/{providerCode}/activate
POST /api/v1/admin/settings/payment-providers/{providerCode}/configure
```

Rules:

- Required permission: `settings.payment.manage`.
- Schema provider-agnostic.
- Tidak boleh expose secret value.
- Hanya satu active provider per workspace per mode.

## 17. Webhook API

Endpoint:

```http
POST /api/v1/webhooks/payments/{providerCode}
```

Rules:

- Verify provider signature before mutation.
- Store raw event or payload hash.
- Process idempotently.
- Duplicate webhook returns safe no-op.
- Webhook accepted does not automatically mean payment is paid.
- Payment status transition remains owned by service/state machine.

## 21. Pagination Contract

List endpoints use:

```http
?page=1&page_size=20
```

OpenAPI schema:

```yaml
PaginationMeta:
  type: object
  properties:
    page:
      type: integer
    page_size:
      type: integer
    total:
      type: integer
    total_pages:
      type: integer
```

Endpoint that needs pagination:

- `GET /api/v1/admin/orders`.
- `GET /api/v1/admin/qr-codes`.
- `GET /api/v1/admin/settings/payment-providers`.

## 22. API Validation Rules

OpenAPI must validate:

- Request body.
- Path params.
- Query params.
- Response schema.
- Enum values.
- Required fields.
- Nullable fields.
- UUID format.
- Date-time format.
- Minimum and maximum values.

Example:

```yaml
quantity:
  type: integer
  minimum: 1
  maximum: 99
```

## 23. Contract Testing

Minimum contract tests:

- Backend response matches OpenAPI schema.
- Invalid request returns documented error.
- Missing required field returns `VALIDATION_ERROR`.
- Unauthorized admin request returns `401`.
- Forbidden admin request returns `403`.
- Webhook invalid signature returns `401` or `403`.
- Generated frontend client compiles.

Recommended tools can be chosen later, such as Redocly CLI, Spectral, Schemathesis, Dredd, Prism, or OpenAPI TypeScript tooling.

## 24. Mock Server Strategy

OpenAPI can generate mock server scenarios for:

- QR universal success.
- QR outlet success.
- QR location success.
- Checkout success.
- Checkout failed product unavailable.
- Payment pending.
- Payment paid.
- Payment expired.
- Admin order ready.
- Webhook accepted.

Frontend and QA may use mock responses before all runtime endpoints are finalized.

## 25. Generated Client Strategy

Target generated clients:

- Frontend web client.
- Admin dashboard client.
- Optional internal worker client.

Generated client must not bypass runtime auth, permission, or validation. It only improves type safety and request/response consistency.

## 26. OpenAPI Governance

Rules:

- Every new endpoint must update `docs/api/openapi.yaml`.
- Breaking change requires API version bump.
- Non-breaking additive fields can stay in `/api/v1`.
- Schema must not expose internal DB fields by accident.
- New error code must be added to `ErrorCode`.
- New enum value must be reviewed.
- Deprecated endpoint must include `deprecated: true`.

## 27. OpenAPI CI Checklist

Every PR that changes API should run:

- OpenAPI lint.
- Schema validation.
- Breaking change detection.
- Generated client compile check.
- Contract test.
- Example payload validation.
