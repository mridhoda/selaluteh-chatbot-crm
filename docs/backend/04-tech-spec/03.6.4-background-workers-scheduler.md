# Phase 3.6.4 - Background Workers & Scheduler

## Objective

Dokumen ini mendefinisikan arsitektur background jobs, scheduler, queue, retry, dan asynchronous processing untuk backend SELKOP. Tujuannya adalah memastikan proses yang tidak perlu berjalan langsung di HTTP request dapat diproses secara aman, idempotent, observable, dan scalable.

Contoh proses yang masuk background worker:

- Expire QR session.
- Expire checkout session.
- Expire payment.
- Payment reconciliation.
- Webhook retry/processing.
- Notification retry.
- Analytics aggregation.
- Cache invalidation.
- Housekeeping cleanup.
- Provider health check.

## Implementation Status

Phase ini sudah memiliki fondasi runtime di codebase:

- `server/src/workers/job-contract.js` mendefinisikan job type, payload contract, backoff helper, error sanitizer, dan wrapper observability dasar.
- `server/src/services/job-queue.service.js` menyediakan abstraksi enqueue/process job berbasis repository.
- `server/src/workers/checkout-cleanup.worker.js` menjalankan expiry checkout sederhana.
- `server/src/workers/qr-session-expiry.worker.js` menjalankan expiry QR session lewat service layer.
- `server/src/workers/payment-reconciliation.worker.js` menjalankan rekonsiliasi payment berkala.
- `server/src/workers/notification.worker.js` memproses delivery notifikasi pending.
- `server/src/workers/telegram-webhook-events.worker.js` memproses inbound webhook event secara asynchronous.
- `server/src/workers/index.js` menjadi registry worker untuk mode in-process MVP.
- `server/src/index.js` men-start scheduler/worker saat server bootstrap.

Status ini belum berarti semua hardening production sudah selesai. Beberapa worker legacy masih memakai direct Supabase update dan perlu bertahap dipindahkan ke service contract/state machine penuh sebelum multi-instance production.

## 1. Background Worker Principles

Prinsip utama:

- HTTP request harus cepat.
- Proses berat dipindahkan ke worker.
- Worker harus idempotent.
- Worker tidak boleh melanggar state machine.
- Worker tidak boleh bypass service layer.
- Worker wajib membuat audit/event untuk perubahan penting.
- Worker wajib observable.
- Worker wajib aman dijalankan ulang setelah crash/restart.

Worker tetap harus mengikuti:

- Domain invariants.
- State machine.
- Service contracts.
- Security rules.
- Transaction boundaries.
- Workspace scope.

Flow yang benar:

```txt
Scheduler / Queue Worker
  -> Application Service
  -> Repository
  -> Database
```

Flow yang ditolak:

```txt
Worker
  -> bebas UPDATE table lintas domain
```

Direct repository call hanya boleh untuk operasi housekeeping sederhana yang tidak mengubah state bisnis kompleks. Untuk payment, checkout, order, dan webhook, worker harus memakai service contract.

## 2. Scheduler vs Queue Worker

Scheduler memicu pekerjaan berdasarkan waktu. Contoh:

- Setiap 1 menit mencari checkout/payment expired.
- Setiap 5 menit menutup QR session expired.
- Setiap 5 menit melakukan provider health check.
- Setiap 1 jam melakukan analytics aggregation.
- Setiap malam membersihkan data temporary lama.

Queue worker memproses job yang sudah masuk queue. Contoh:

- `notification.retry`.
- `webhook.process`.
- `payment.reconcile`.
- `checkout.expire_sessions`.

Runtime MVP masih memakai scheduler in-process berbasis `setInterval`. Untuk job yang perlu durability dan retry lintas crash, gunakan `job-queue.service.js` dan repository jobs.

## 3. Architecture Overview

```txt
HTTP Controller / Webhook Controller
  -> validate quickly
  -> persist event or enqueue job
  -> return response

Scheduler
  -> enqueue due jobs or run bounded scan

Queue Worker
  -> claim job
  -> Application Service
  -> Repository
  -> Database
  -> mark complete / retry / dead letter
```

Untuk MVP single-instance, API dan worker boleh berjalan dalam proses Node yang sama. Untuk growth/production, pisahkan menjadi minimal dua runtime:

```txt
backend-api
backend-worker
queue/storage backend
```

## 4. Job Categories

| Category | Purpose | Example |
|---|---|---|
| Time-based jobs | Berjalan berdasarkan waktu | Expire checkout, expire QR, cleanup idempotency key |
| Event-based jobs | Muncul dari event bisnis | Send notification after payment success |
| Provider-based jobs | Sinkronisasi provider eksternal | Payment reconciliation, webhook processing, provider health check |
| Analytics jobs | Membuat data summary/report | QR conversion, daily sales, product sales |
| Housekeeping jobs | Membersihkan data temporary | Old sessions, old raw payload refs, temporary cache |

## 7. Checkout Expiry Worker

Purpose: menandai checkout session yang sudah lewat masa aktif.

Flow:

```txt
Find checkout_sessions
where status in ('draft', 'validated')
and expires_at < now()
  -> CheckoutService.expireCheckout()
  -> update checkout status
  -> audit/event
```

Rule:

- Checkout yang sudah converted menjadi order tidak boleh di-expire.
- Worker harus memakai atomic status guard.
- Expiry harus idempotent by checkout ID and target status.

Runtime note: `checkout-cleanup.worker.js` sudah melakukan expiry sederhana via repository. Hardening berikutnya adalah memindahkan transisi penting ke `CheckoutService.expireCheckout()`.

## 8. QR Session Expiry Worker

Purpose: menutup QR session yang sudah melewati TTL.

Flow:

```txt
Find qr_sessions
where status = active
and expires_at < now()
  -> QrSessionService.expireSession()
  -> mark expired
  -> track analytics
```

Rule:

- QR session expired tidak boleh digunakan untuk checkout baru.
- Worker wajib menjaga workspace/outlet scope.
- Worker tidak boleh menciptakan checkout/order baru.

Runtime note: `qr-session-expiry.worker.js` memanggil `expireQrSessions()` dari service layer dan sudah terdaftar saat bootstrap.

Implementation note Phase 4 section 10: cleanup menandai QR order session lama sebagai `session_status='expired'`, `is_active=false`, dan `revoked_at=now`. Row tidak dihapus sehingga histori operasional/order link tetap tersedia.

## 9. Payment Expiry Worker

Purpose: menandai payment pending/processing yang sudah melewati `expires_at`.

Flow:

```txt
Find payments
where status in ('pending', 'processing')
and expires_at < now()
  -> optional provider status check outside DB transaction
  -> PaymentService.expirePayment()
  -> order public status becomes payment_expired
```

Rule:

- Worker tidak boleh mengubah payment menjadi `paid` tanpa webhook valid atau provider verification.
- Expiry harus atomic dan hanya berlaku dari status non-terminal yang diizinkan.
- Jika provider menyatakan paid tetapi amount/currency mismatch, payment masuk `manual_review`.

Runtime implementation Phase 4 section 10:

- `paymentsRepository.findDueForExpiry()` mencari payment `pending`/`processing` dengan `expires_at <= backend now`.
- `payment-expiry.service.js` melakukan transisi via `transitionStatus()` dengan guard status `pending`/`processing`, sehingga `paid` tidak bisa ikut expired.
- Worker menjalankan expiry berkala dari backend process; tidak ada dependency ke timer frontend.

## 10. Payment Reconciliation Worker

Purpose: menangani webhook telat, hilang, gagal, atau payment yang stuck pending.

Flow:

```txt
Find payments pending/processing in reconciliation window
  -> PaymentProviderResolver
  -> Adapter.getPayment()
  -> normalize provider status
  -> validate amount/currency/reference
  -> apply payment state transition
```

Use case:

- Customer sudah bayar tetapi webhook tidak masuk.
- Provider timeout saat callback.
- Duplicate webhook gagal diproses.
- Payment status stuck pending.

Rule:

- Provider call dilakukan sebelum atau di luar DB transaction.
- Satu payment tidak boleh direconcile oleh dua worker bersamaan.
- Paid transition harus melewati service/state machine.
- Mismatch amount/currency/reference masuk `manual_review` dan wajib audit trail.

Runtime implementation Phase 4 section 10:

- Worker mengambil payment `missing_webhook` melalui `detectMissingWebhooks()` lalu memanggil `reconcileMissingWebhook()`.
- Worker juga memeriksa payment `pending`/`processing` yang cukup lama lewat `reconcilePendingProviderPayment()`.
- Service memakai `resolvePaymentProvider(..., capability: 'statusQuery')`, sehingga provider yang mendukung status query memakai adapter terkonfigurasi; provider tanpa dukungan/config gagal aman dan hanya dilog.
- Reconciliation audit dicatat ke `reconciliation_audit`.
- Paid notification tidak dikirim ulang jika order sudah berada di status paid sebelum proses reconciliation.

## 11. Webhook Event Processor Worker

Purpose: memproses webhook secara cepat, aman, dan idempotent.

Recommended flow:

```txt
Webhook received
  -> verify signature or channel token
  -> store raw event / payload hash
  -> enqueue webhook.process job
  -> return 2xx quickly
  -> worker parse payload
  -> normalize status/event
  -> apply service-layer transition
  -> publish domain event
```

Duplicate webhook key:

```txt
provider + provider_reference/provider_transaction_id + provider_event_id
```

Duplicate behavior:

- Return success/no-op.
- Do not duplicate payment event.
- Do not duplicate audit log.
- Do not send duplicate notification.

## 12. Notification Retry Worker

Purpose: mengirim ulang notifikasi yang gagal atau tertunda.

Examples:

- Payment success.
- Order accepted.
- Order ready.
- Order completed.
- Payment failed/expired.

Rule:

- Notification failure tidak boleh menggagalkan order.
- Delivery key harus idempotent.
- Retry harus bounded dan masuk failed/dead-letter setelah max attempt.

Runtime note: `notification.worker.js` memproses `notification_deliveries` pending. Hardening berikutnya adalah retry backoff dan permanent failure/dead-letter yang lebih eksplisit.

## 13. Analytics Aggregation Worker

Purpose: membuat data dashboard lebih cepat dibaca.

Aggregate target:

- QR scans.
- QR conversion.
- Outlet sales.
- Payment success rate.
- Product sales.
- Peak hour.
- Checkout drop-off.

Rule:

- Analytics worker hanya membaca immutable event/order/payment data.
- Analytics worker tidak boleh mengubah order/payment business state.
- Recompute harus idempotent by `(workspace_id, metric_key, period_start, period_end)`.

## 14. Cleanup Worker

Purpose: membersihkan data temporary yang sudah lewat retention.

Target cleanup:

- Expired checkout sessions.
- Expired QR sessions.
- Expired idempotency keys.
- Old security events.
- Old webhook raw payload refs.
- Temporary cache entries.

Rule:

- Tidak boleh hard delete `orders`, `payments`, `audit_logs`, `payment_history`, atau `order_history`.
- Cleanup harus bounded per run agar tidak mengunci database terlalu lama.
- Hard delete hanya untuk data temporary/non-ledger yang retention policy-nya eksplisit.

## 15. Provider Health Check Worker

Purpose: memantau health provider seperti BayarGG/Xendit/DOKU.

Flow:

```txt
Every 5 minutes
  -> PaymentProviderService.checkHealth()
  -> Adapter.healthCheck()
  -> record provider health status
  -> alert if degraded/down
```

Status provider:

- `healthy`.
- `degraded`.
- `down`.
- `unknown`.

Use case:

- Provider timeout.
- Provider API down.
- Webhook endpoint bermasalah.
- Create payment gagal berulang.

## 16. Job Payload Contract

Canonical payload mengikuti `server/src/workers/job-contract.js`:

```json
{
  "jobId": "uuid",
  "jobType": "payment.reconcile",
  "workspaceId": "uuid",
  "entityType": "payment",
  "entityId": "payment_id",
  "attempt": 1,
  "createdAt": "timestamp",
  "scheduledAt": "timestamp",
  "metadata": {}
}
```

Rules:

- Payload jangan terlalu besar.
- Simpan ID entity saja.
- Worker mengambil data terbaru dari database.
- Jangan simpan secret/raw credential di payload.
- Gunakan stable deduplication key untuk enqueue idempotent.

## 17. Job Idempotency

Semua job penting harus aman jika dijalankan lebih dari sekali.

Gunakan kombinasi:

- `jobId`.
- `deduplicationKey`.
- `workspaceId`.
- `entityType` and `entityId`.
- Current status check.
- Unique event key.
- Atomic update guard.

Example:

```txt
ExpirePaymentJob dijalankan 3 kali
  -> status guard hanya mengubah pending/processing sekali
  -> payment history tidak duplicate
  -> audit/event tidak duplicate
```

## 18. Retry Strategy

Default retry:

| Job Type | Retry | Backoff |
|---|---:|---|
| Webhook processing | 5x | Exponential |
| Payment reconciliation | 3x | Exponential |
| Notification | 5x | Exponential |
| Analytics | 3x | Linear |
| Cleanup | 1-3x | Linear |
| Provider health check | No retry; next schedule | Scheduled |

Backoff baseline:

```txt
1st retry: 30 seconds
2nd retry: 2 minutes
3rd retry: 5 minutes
4th retry: 15 minutes
5th retry: 30 minutes
```

Runtime note: `computeWorkerBackoffMs()` supports exponential/linear backoff. `job-queue.service.js` delegates queue retry scheduling to `retry-policy.js`.

## 19. Dead Letter Queue

Job yang gagal permanen masuk ke dead letter storage.

Minimal fields:

- `job_id`.
- `job_type`.
- `workspace_id`.
- `entity_type`.
- `entity_id`.
- `last_error_code`.
- `last_error_message`.
- `attempt_count`.
- `payload_json`.
- `failed_at`.
- `resolved_at`.

Admin/internal tooling harus bisa melihat failed webhook processing, failed notification, failed reconciliation, dan provider error.

## 20. Locking & Concurrency

Worker bisa berjalan paralel, jadi locking wajib.

Strategi:

- `SELECT ... FOR UPDATE SKIP LOCKED` untuk queue claim.
- Advisory lock untuk singleton/batch domain tertentu.
- Unique processing key.
- Job status lock.
- Distributed lock jika multi-instance.

Rules:

- Satu payment tidak boleh direconcile oleh dua worker bersamaan.
- Satu checkout tidak boleh di-expire dua worker bersamaan.
- Satu webhook event tidak boleh diproses dua kali.
- Worker scan harus bounded dan tidak mengambil semua row sekaligus.

## 21. Transaction Boundaries

Contoh payment reconciliation:

```txt
Provider query outside DB transaction
  -> begin transaction
  -> lock payment row
  -> validate current status
  -> validate amount/currency/reference
  -> apply transition
  -> insert payment status history
  -> insert audit log
  -> commit
  -> publish notification event
```

Rules:

- External provider call dilakukan sebelum transaction atau di luar transaction.
- Database transaction tidak boleh menunggu external API.
- Post-commit side effect seperti notification harus bisa retry terpisah.

- `job_id`.
- `job_type`.
- `workspace_id`.
- `entity_id`.
- `attempt`.
- `duration_ms`.
- `status`.
- `error_code`.

Metrics:

- `jobs_processed_total`.
- `jobs_failed_total`.
- `job_duration_ms`.
- `retry_count`.
- `dead_letter_count`.
- `queue_lag_seconds`.
- `worker_heartbeat`.

Runtime note: `runWorkerJob()` already captures duration and sanitized failure output. Production hardening should move console logs to structured logger/metrics sink.

## 23. Worker Security

Rules:

- Worker hanya berjalan dari trusted runtime.
- Worker tidak memiliki public endpoint.
- Worker memakai service account internal.
- Worker tetap enforce workspace scope.
- Worker tidak boleh expose secret di log.
- Worker tidak boleh mencetak raw webhook secret.
- Worker tidak boleh memakai payload user tanpa re-validasi entity terbaru dari DB.

## 24. Suggested Tables

Jika queue belum memakai managed queue seperti Redis/BullMQ/SQS, mulai dengan Postgres-backed jobs.

Alpha tables:

- `background_jobs` or current `jobs` table.
- `dead_letter_jobs`.
- `worker_heartbeats`.

Growth tables:

- `provider_health_checks`.
- `notification_jobs` or `notification_deliveries`.
- `analytics_snapshots`.

## 25. Recommended Queue Technology

Recommendation:

| Stage | Queue Technology |
|---|---|
| Alpha | Postgres-based scheduler + simple worker |
| Growth | BullMQ + Redis |
| Scale | RabbitMQ, Pub/Sub, SQS, Kafka, or Redis Streams |

Decision rule:

- Keep Postgres queue while volume is low and operational simplicity matters.
- Move to BullMQ/Redis when retry volume, delayed jobs, or worker concurrency need clearer operational control.
- Move to broker/stream when cross-service eventing and high throughput become core requirements.
