# Phase 3.6.3 - External Provider Architecture

## Objective

Dokumen ini mendefinisikan arsitektur integrasi external provider agar backend SELKOP tidak terkunci ke satu vendor. Provider aktif saat alpha dapat berupa BayarGG, tetapi core checkout/order/payment harus tetap provider-agnostic dan siap untuk Xendit, DOKU, Midtrans, Manual Transfer, atau provider lain.

## Implementation Status

Phase ini sudah diterapkan di runtime code, bukan hanya dokumentasi:

- `server/src/integrations/payments/payment-provider.types.js` diperluas dengan provider code, method, status normalizer, safe error, dan adapter contract assertion.
- `server/src/integrations/payments/provider-capabilities.js` menampung capability matrix provider.
- `server/src/integrations/payments/payment-adapter-registry.js` menjadi registry adapter provider.
- `server/src/services/payment-provider-resolver.service.js` menjadi resolver provider aktif berbasis workspace, mode, config, method, dan capability.
- `server/src/services/payment.service.js` tidak lagi hardcode dynamic import adapter di service.
- `server/src/services/payment-webhook.service.js` memakai resolver/registry untuk adapter webhook provider.

## 1. External Provider Scope

Scope phase ini:

- Payment provider.
- Webhook provider.
- Payment status verification.
- Refund provider.
- Future notification provider.
- Future delivery provider.

Alpha focus:

- Payment provider architecture.
- BayarGG adapter.
- Webhook verification.
- Payment reconciliation.
- Provider config.

## 2. Provider-Agnostic Principle

Core payment schema and service must not store provider-specific field names like:

```txt
bayargg_invoice_id
xendit_invoice_id
midtrans_token
```

Use neutral fields:

```txt
provider_transaction_id
provider_ref
merchant_reference
payment_url
provider_status
metadata.raw_provider_response
```

Provider-specific data may exist only inside adapter-normalized metadata, never as business logic dependency in order/checkout core.

## 3. Provider Architecture Overview

Runtime flow:

```txt
CheckoutService / PaymentService
  -> PaymentProviderResolver
  -> PaymentAdapterRegistry
  -> PaymentAdapter Interface
     -> BayarGGAdapter
     -> DokuAdapter
     -> XenditAdapter
     -> MidtransAdapter
```

Webhook flow:

```txt
Webhook Controller
  -> PaymentWebhookService
  -> PaymentProviderResolver / AdapterRegistry
  -> Adapter.verifyWebhook()
  -> Normalize event
  -> PaymentRepository
  -> OrderService/state transition
```

## 4. PaymentProviderService Responsibility

Payment orchestration must:

- Resolve active provider.
- Create payment.
- Verify provider response.
- Normalize provider status.
- Handle webhook.
- Trigger payment state transition.
- Record payment events.

It must not know provider API details such as BayarGG endpoint path or Xendit auth header.

## 5. PaymentProviderResolver

Runtime module: `server/src/services/payment-provider-resolver.service.js`.

Resolver selects provider from:

- `workspaceId`.
- mode/environment.
- active provider setting.
- requested provider override.
- payment method.
- provider capability.

Example:

```txt
workspace = SELKOP
mode = sandbox
active provider = bayargg
method = qris
resolver returns BayarGG adapter + runtime config
```

## 6. Payment Adapter Interface

All providers must implement:

```ts
createPayment(input, config)
verifyWebhook(rawBody, headers, config)
getPayment(providerTransactionId, config)
```

Optional aliases/defaults:

```ts
createPaymentSession(input, config)
getPaymentSession(providerTransactionId, config)
cancelPayment(providerTransactionId, config)
refundPayment(providerTransactionId, amount, config)
```

The runtime assertion is implemented by `assertPaymentAdapterContract()`.

## 7. Provider Capability Matrix

Runtime module: `server/src/integrations/payments/provider-capabilities.js`.

| Provider | QRIS | VA | E-Wallet | Card | Refund | Webhook | Status Query |
|---|---:|---:|---:|---:|---:|---:|---:|
| BayarGG | Yes | Yes | Yes | No | No | Yes | Yes |
| DOKU | Yes | Yes | Yes | Yes | Yes | Yes | No |
| Xendit | Yes | Yes | Yes | Yes | Yes | Yes | Yes |
| Midtrans | Yes | Yes | Yes | Yes | Yes | Yes | Yes |
| Manual Transfer | No | Yes | No | No | Manual | No | No |

Backend must check capability before invoking provider-specific behavior.

## 8. Provider Configuration

Provider config is resolved from:

- `payment_provider_settings` where available.
- runtime settings/environment fallback via `getPaymentRuntimeConfig()`.

Rule:

```txt
Only one active provider per workspace per mode.
```

Existing payments keep their original provider even after provider switching.

## 9. Secret Management

Secrets must not be stored as plain text in DB.

Allowed sources:

- Environment variables.
- Secret manager.
- Vault.
- Encrypted runtime config.

Logs must never include:

- API key.
- Secret key.
- Webhook secret.
- Authorization header.
- Full raw customer sensitive payload.

## 10. Create Payment Flow

```txt
CheckoutService
  -> create order + payment record intent
  -> commit DB transaction
  -> PaymentService.createPaymentSessionForOrder()
  -> PaymentProviderResolver.resolvePaymentProvider()
  -> Adapter.createPaymentSession()
  -> PaymentRepository stores provider reference and URL
```

External API calls must not run inside database transaction.

## 11. Webhook Flow

```txt
Provider Webhook
  -> Webhook Controller
  -> PaymentWebhookService
  -> Adapter.verifyWebhook()
  -> Normalize status
  -> Idempotency check by provider event ID
  -> Payment transition
  -> Order transition
  -> Audit/notification
```

Webhook body is untrusted until verified by adapter/provider-specific strategy.

## 12. Status Normalization

All provider statuses must map to internal enum:

```txt
unpaid
pending
processing
paid
failed
expired
refunded
cancelled
manual_review
```

Runtime helper:

- `normalizeProviderStatus()` in `payment-provider.types.js`.

## 13. Webhook Idempotency

Webhook processing key:

```txt
provider + provider_reference/provider_transaction_id + provider_event_id
```

Duplicate webhook behavior:

- Return success/no-op response.
- Do not update payment again.
- Do not duplicate payment event.
- Do not duplicate audit log.

## 14. Payment Reconciliation

Reconciliation flow:

```txt
Pending payment
  -> Scheduler/manual sync
  -> PaymentProviderResolver
  -> Adapter.getPayment()
  -> Normalize status
  -> Update payment/order if changed
```

Use cases:

- Delayed webhook.
- Failed webhook.
- Provider timeout.
- Customer already paid but local state still pending.

## 15. Error Handling

Provider errors must be normalized to safe app errors.

Provider categories:

- `PROVIDER_UNAVAILABLE`.
- `PROVIDER_TIMEOUT`.
- `PROVIDER_INVALID_RESPONSE`.
- `PROVIDER_UNAUTHORIZED`.
- `PROVIDER_RATE_LIMITED`.
- `PROVIDER_PAYMENT_REJECTED`.
- `PROVIDER_SIGNATURE_INVALID`.
- `PROVIDER_AMOUNT_MISMATCH`.
- `PROVIDER_REFERENCE_NOT_FOUND`.

Safe frontend errors:

- `PAYMENT_PROVIDER_UNAVAILABLE`.
- `PAYMENT_CREATION_FAILED`.
- `PAYMENT_STATUS_UNKNOWN`.
- `PAYMENT_EXPIRED`.

## 16. Manual Review Rules

Payment enters `manual_review` when:

- Amount mismatch.
- Currency mismatch.
- Unknown provider reference.
- Invalid status transition.
- Duplicate suspicious webhook.
- Signature valid but payload inconsistent.
- Provider says paid but order amount differs.

Manual review must create payment event and audit trail.

## 17. Refund Architecture

Refund interface is prepared but optional for alpha:

```ts
refundPayment(providerTransactionId, amount, config)
```

Refund must not delete order. It must emit/record payment refund and audit events.

## 18. Provider Switching

Provider switch rules:

- Requires settings/payment management permission.
- Creates audit log.
- Existing payments keep old provider.
- New payments use the new active provider.
- Old order/payment records are not automatically migrated.

## 19. Observability

Provider call logs must include safe fields:

- `provider_code`.
- `workspace_id`.
- `payment_id`.
- `order_id`.
- `request_id`.
- `provider_reference`.
- `duration_ms`.
- `status`.
- `error_code`.

Never log secrets or full sensitive raw payloads.

## 20. Testing Strategy

Required tests:

- Adapter contract assertion.
- Provider capability matrix.
- Provider resolver active provider resolution.
- Webhook signature valid/invalid.
- Duplicate webhook.
- Amount mismatch.
- Payment reconciliation.
- Provider switching.

## 21. Future Provider Extension

To add a provider:

1. Add provider setting record/config.
2. Create adapter implementing contract.
3. Register adapter in `payment-adapter-registry.js`.
4. Add capability entry in `provider-capabilities.js`.
5. Add status mapping.
6. Add webhook route/handler if needed.
7. Add tests.

Do not change checkout/order core for a new provider.
