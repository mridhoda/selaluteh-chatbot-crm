# Phase 3.6.1 - Service Contracts

## Objective

Dokumen ini mendefinisikan kontrak setiap backend service agar komunikasi antar-service konsisten, terdokumentasi, type-safe, mudah diuji, dan tetap mematuhi QR domain invariants serta domain events/state machines.

Dokumen ini menjadi acuan implementasi Application Service. Controller, worker, AI tool executor, dan webhook handler harus memanggil service melalui contract, bukan langsung membaca repository atau entity database lintas domain.

## Relationship To Previous Phases

- `03.5-qr-domain-architecture.md` mendefinisikan QR business domain dan invariants.
- `03.5.1-domain-events-state-machines.md` mendefinisikan domain events, state machines, allowed transitions, retry, idempotency, dan payload contracts.
- `03.6.1-service-contracts.md` mendefinisikan operasi service, DTO, result, error, authorization, idempotency, ownership, dan transaction boundary.

## 1. Service Contract Principles

Service contract adalah boundary resmi untuk business operation.

Principles:

- Service hanya expose business operation, bukan database operation mentah.
- Controller tidak mengetahui repository.
- Repository tidak mengetahui service lain.
- Service tidak membaca database domain lain secara langsung.
- Semua komunikasi antar-domain melalui service contract.
- Entity database tidak boleh keluar dari service sebagai response internal.
- Semua service wajib menerima workspace context untuk tenant-owned operation.
- Semua operation yang mengubah money, payment, order, QR, session, atau fulfillment wajib menghasilkan audit/event sesuai kebutuhan.

Allowed shape:

```txt
Controller / Worker / Webhook Handler / AI Tool
  -> Application Service Contract
  -> Repository owned by that service
  -> Result DTO
```

Rejected shape:

```txt
Controller
  -> Repository
  -> Entity
  -> Response
```

Rejected cross-domain shape:

```txt
OrderService
  -> ProductRepository
  -> PaymentRepository
```

Correct cross-domain shape:

```txt
CheckoutService
  -> CartValidationService
  -> PaymentProviderService
  -> OrderService
```

## 2. Contract Design Standards

Every service operation must document:

- Purpose.
- Input DTO.
- Output DTO.
- Possible errors.
- Authorization requirement.
- Idempotency requirement.
- Transaction behavior.
- Domain events emitted.
- Side effects.

Standard operation flow:

```txt
Input DTO
  -> Input validation
  -> Authorization check
  -> Business invariant validation
  -> Business logic
  -> Repository write/read owned by service
  -> Domain events / audit / side effects
  -> Result DTO
```

Contract naming:

- Operation names use verb + domain object: `resolveQrCode`, `createCheckout`, `acceptOrder`.
- DTO names use operation + request/result: `CreateCheckoutRequest`, `CreateCheckoutResult`.
- Error codes use uppercase snake case: `QR_EXPIRED`, `DUPLICATE_CHECKOUT`.
- Money is always integer minor unit for IDR.

## 4. Common Result Objects

Service results are internal application results, not HTTP responses.

```ts
type ServiceResult<T> =
  | { ok: true; data: T; events?: DomainEventRef[] }
  | { ok: false; error: ServiceError }

type DomainEventRef = {
  eventId: string
  eventName: string
}
```

Result kinds:

| Kind | Meaning |
|---|---|
| `Success` | Operation completed |
| `Failure` | Operation failed with typed error |
| `ValidationError` | Input or business validation failed |
| `NotFound` | Scoped entity not found |
| `Forbidden` | Actor lacks permission |
| `Conflict` | State conflict or duplicate operation |
| `UnexpectedError` | Unexpected infrastructure/runtime failure |

HTTP mapping is handled by controller/error middleware, not by service contract.

## 5. Common Error Objects

```ts
type ServiceError = {
  code: ServiceErrorCode
  category: 'business' | 'validation' | 'not_found' | 'forbidden' | 'conflict' | 'security' | 'provider' | 'infrastructure'
  message: string
  details?: Record<string, unknown>
  retryable?: boolean
}
```

Common error codes:

| Code | Category | Meaning |
|---|---|---|
| `UNAUTHORIZED_WORKSPACE` | security | Workspace context does not match resource |
| `FORBIDDEN_ROLE` | forbidden | Actor lacks role/permission |
| `INVALID_QR` | business | QR does not exist or scope invalid |
| `EXPIRED_QR` | business | QR has expired |
| `REVOKED_QR` | business | QR has been revoked |
| `EXPIRED_SESSION` | business | QR session expired |
| `DUPLICATE_CHECKOUT` | conflict | Session already produced successful checkout |
| `CLOSED_OUTLET` | business | Outlet is not accepting orders |
| `UNAVAILABLE_PRODUCT` | business | Product is inactive/deleted/unavailable |
| `INVALID_MODIFIER` | validation | Modifier invalid for product |
| `PAYMENT_FAILED` | provider | Provider payment failed |
| `INVALID_PROVIDER` | provider | Provider config invalid or inactive |
| `WEBHOOK_SIGNATURE_INVALID` | security | Webhook verification failed |
| `INVALID_STATE_TRANSITION` | conflict | Transition violates state machine |

Rules:

- Do not use free-form string codes.
- Do not leak raw provider/database error to user-facing response.
- Include `retryable=true` only when retry is safe and idempotent.

### Operations

|---|---|---|

| `getProductDetail` | Return product detail and modifiers | None |
| `resolveOutlet` | Resolve outlet from QR scope or customer selection | None |
| `getAvailableProducts` | Return only products available for outlet | None |

### `resolveStorefront`

```ts
type ResolveStorefrontRequest = WorkspaceContext & {
  qrId?: string
  sessionId?: string
  storefrontId?: string
}
```

Output:

```ts
type StorefrontResolved = {
```

## 11. Checkout Service Contract

Owner service for checkout orchestration and atomic conversion from cart intent to order/payment.

### Operations

| Operation | Purpose | Emits Events |
|---|---|---|
| `startCheckout` | Start checkout flow | `CheckoutStarted` |
| `validateCheckout` | Validate all checkout preconditions | `CheckoutValidated` or `CheckoutRejected` |
| `createCheckout` | Atomically create order/payment/checkout marker | `CheckoutConfirmed` |
| `cancelCheckout` | Cancel checkout before confirmation | `CheckoutExpired` or domain-specific event |
| `expireCheckout` | Expire stale checkout | `CheckoutExpired` |

### `createCheckout`

Input:

```ts
type CreateCheckoutRequest = WorkspaceContext & {
  sessionId: string
  cartId: string
  customerContext: CustomerContext
  outletContext: OutletContext
  fulfillmentType: 'pickup' | 'takeaway' | 'dine_in' | 'delivery'
  paymentProviderId: string
  idempotencyKey: string
}
```

Output:

```ts
type CheckoutCreatedResult = {
  checkoutId: string
  orderId: string
  paymentId: string
  paymentIntent: {
    provider: string
    providerReference: string
    paymentUrl?: string
    expiresAt?: string
  }
  orderDraft: {
    status: 'pending_payment'
    total: Money
  }
}
```

Possible errors:

- `EXPIRED_SESSION`.
- `DUPLICATE_CHECKOUT`.
- `CLOSED_OUTLET`.
- `UNAVAILABLE_PRODUCT`.
- `INVALID_MODIFIER`.
- `INVALID_PROVIDER`.

Transaction: primary owner for checkout transaction.

Transaction scope:

- Validate session not checked out.
- Recalculate cart.
- Create order.
- Create order items.
- Create payment record.
- Mark checkout/session completion state.
- Write outbox events.

No side-effect inside transaction:

- Sending notification.
- External provider call if it cannot be rolled back.
- Analytics network call.

- `INVALID_PROVIDER`.

## 13. Payment Adapter Contract

Every payment provider integration must implement this adapter contract.

```ts
interface PaymentAdapter {
  createPayment(input: AdapterCreatePaymentInput): Promise<AdapterCreatePaymentResult>
  verifyWebhook(input: AdapterVerifyWebhookInput): Promise<AdapterWebhookResult>
  getStatus(input: AdapterGetStatusInput): Promise<AdapterPaymentStatusResult>
  cancel(input: AdapterCancelPaymentInput): Promise<AdapterCancelPaymentResult>
  refund(input: AdapterRefundPaymentInput): Promise<AdapterRefundPaymentResult>
}
```

Provider examples:

- Xendit.
- Midtrans.
- Manual payment.
- Future provider.

Rules:

- Application services must depend on `PaymentAdapter`, not provider SDK directly.
- Adapter returns normalized provider result.
- Adapter must not update order state directly.
- Adapter must not emit notification directly.

## 20. Cross-Service Communication Rules

Allowed:

```txt
CheckoutService
  -> CartValidationService
  -> PaymentProviderService
  -> OrderService
```

Allowed:

```txt
PaymentProviderService
  -> PaymentAdapter
  -> OrderService via PaymentSucceeded event/subscription
```

Rejected:

```txt
Controller
  -> PaymentRepository
```

Rejected:

```txt
OrderService
  -> ProductRepository
  -> PaymentRepository
```

Rules:

- Synchronous calls are allowed for request-time validation/orchestration.
- Domain events are preferred for post-commit side effects.
- Services own their repositories.
- No service may bypass another service's state machine.
- No service may update another domain's table directly unless it is explicitly the transaction owner for that aggregate.

## 21. Versioning Strategy

Contract versioning rules:

- Additive fields are backward compatible.
- Removing or changing required fields requires a new version.
- Error code semantics must not change without version bump.
- Consumers must tolerate unknown optional fields.
- Service contract v1 remains stable during MVP.

Version example:

```ts
type CreateCheckoutRequestV1 = { /* stable MVP contract */ }
type CreateCheckoutRequestV2 = { /* future additive/breaking changes */ }
```

## 22. Error Handling Contract

All service operations return `ServiceResult<T>` or throw only infrastructure-level exceptions caught at service boundary and converted to `ServiceError`.

Error categories:

- Business error.
- Validation error.
- Infrastructure error.
- Security error.
- Provider error.
- Conflict error.

Rules:

- Do not throw random strings or raw provider errors.
- Do not map errors to HTTP inside service.
- Controller maps service result to HTTP response.
- Worker maps service result to retry/dead-letter behavior.
- AI tool executor maps service result to safe user-facing answer.

## 23. Transaction Contract

Transaction ownership:

| Service | Transaction Behavior |
|---|---|
| `CheckoutService` | Primary owner for checkout/order/payment marker atomic transaction |
| `OrderService` | Owner for order state transition transaction |
| `PaymentProviderService` | Owner for payment status update transaction |
| `QrSessionService` | Owner for session create/expire/close transaction |
| `CartService` | Owner for cart mutation transaction |
| `AuditService` | Owns audit append transaction |
| `NotificationService` | No business transaction owner |
| `AnalyticsService` | No business transaction owner |
| `CartValidationService` | Read-only, no transaction owner |
| `SecurityService` | Read-only or policy write only, no business transaction owner |

Rules:

- Only one service should own a transaction boundary per business operation.
- Collaborating services may join transaction only through explicit transaction context.
- External network calls should not run inside DB transaction unless unavoidable and safe.
- Outbox event writes should happen in the same transaction as the state change.

Example:

```txt
CheckoutService
  BEGIN TRANSACTION
    validate session
    calculate backend price
    create order
    create order items
    create payment record
    mark checkout/session state
    write outbox events
  COMMIT
  post-commit: payment provider call / notification / analytics
```

## 24. Future Compatibility

Service contracts must remain compatible with:

- Microservices.
- Event bus.
- GraphQL.
- gRPC.
- Mobile apps.
- AI agent tools.
- Public API.
- Multi-brand.
- Delivery.
- Loyalty and membership.

Rules:

- Contract DTOs should be serializable.
- Avoid framework-specific types in service signatures.
- Avoid leaking ORM/database entity shape.
- Keep provider integrations behind adapter contracts.
- Keep AI tool calls behind the same service contracts as UI/API calls.
