# Phase 3.6.2 - Repository Architecture

## Objective

Dokumen ini mendefinisikan pola akses database yang aman, konsisten, reusable, dan sesuai domain boundary QR Store. Repository adalah lapisan data access. Repository tidak menyimpan business logic, tidak memanggil provider eksternal, dan tidak mengambil keputusan bisnis.

Implementasi runtime terkait phase ini berada di `server/src/db/repositories/`.

## Implementation Status

Phase ini tidak berhenti di dokumentasi. Fondasi repository architecture sudah diterapkan di kode:

- `server/src/db/repositories/repository-contract.js`
- `server/src/db/repositories/idempotency.supabase.repository.js`
- `server/src/db/repositories/analytics.supabase.repository.js`
- Aggregate aliases di `server/src/db/repositories/index.js`
- Method tambahan di repository storefront, outlet, catalog/product, checkout, QR session, dan payment provider settings.
- Public checkout idempotency dipindahkan dari raw service query ke `idempotencyRepository`.

## 1. Repository Principles

Repository principles:

- Controller tidak boleh akses database langsung.
- Service tidak boleh menulis raw SQL/query builder berulang-ulang untuk domain persistence.
- Repository tidak boleh memanggil service lain.
- Repository tidak boleh memanggil payment provider.
- Repository tidak boleh mengandung business decision.
- Repository hanya mengambil, menyimpan, mengubah, dan mengunci data.
- Semua tenant-owned query wajib scoped by `workspace_id`.

Flow yang benar:

```txt
Controller / Worker / Webhook Handler
  -> Service
  -> Repository
  -> Database
```

Flow yang ditolak:

```txt
Controller
  -> Database
```

Flow yang ditolak:

```txt
Repository
  -> Payment Provider
```

## 4. Workspace Scoping Rule

All business repository methods must receive `workspaceId` unless they are explicitly public bootstrap lookups.

Required pattern:

```js
findOrderById({ workspaceId, orderId })
findProductsByOutlet({ workspaceId, outletId })
findActiveProviderSettings({ workspaceId, mode })
```

Rejected pattern:

```js
findOrderById(orderId)
```

Public bootstrap exceptions:

- QR token lookup to resolve workspace.
- Payment provider callback lookup by provider reference.
- Public order token lookup.

After bootstrap resolution, all subsequent repository calls must use `workspaceId`.

## 5. Command vs Query Repository

Query methods read data:

- `findById`
- `findBySlug`
- `listActiveProducts`
- `findOrderDetail`

Command methods write/update data:

- `createOrder`
- `updatePaymentStatus`
- `markQrSessionCompleted`
- `appendAuditLog`

Complex dashboard reads should use filter/specification objects instead of many narrowly named methods.

## 6. Repository Return Type

Repository must not return raw database rows directly to controllers.

Preferred return values:

- Internal DTO.
- Aggregate object.
- Domain-shaped object mapped to camelCase.

Example:

```ts
type OrderAggregate = {
  order: Order
  items: OrderItem[]
  payment?: Payment
  qrContext?: QrContext
}
```

Runtime convention uses `mapRow` and `mapRows` from `server/src/db/supabase-mapper.js`.

## 7. Transaction-Aware Repository

Repository supports transaction-aware wrapping through `withTx` where needed.

Shared helpers:

- `withRepositoryTx(repository)`
- `createTxRepository(repository, tx)`
- `normalizeRepositoryError(error, context)`

Pattern:

```js
const orderRepo = ordersRepository.withTx(tx)
const paymentRepo = paymentsRepository.withTx(tx)
```

Current Supabase JS repositories still primarily use service-role client. For multi-table atomic operations, services should own transaction boundaries via `withTransaction` or RPC. Repository `withTx` support is the contract boundary for future hardening.

## 8. Locking Strategy

Operations requiring row lock:

| Operation | Lock Target |
|---|---|
| Checkout with idempotency | `order_idempotency_records` |
| Payment webhook processing | `payments` |
| Admin order status update | `orders` |
| QR session completion | `qr_order_sessions` |
| Product availability update | `product_outlet_availability` |

Current Supabase JS cannot issue `SELECT FOR UPDATE` directly. Locking should be implemented through transaction/RPC hardening for these methods:

- `idempotencyRepository.findByKeyForUpdate`
- `paymentsRepository.findPaymentForUpdate`
- `ordersRepository.findOrderForUpdate`

## 10. Query Specification Pattern

Use filter objects for complex admin dashboard queries.

Example:

```ts
type OrderListFilter = {
  workspaceId: string
  outletIds?: string[]
  channel?: 'online_store' | 'qr_store'
  paymentStatus?: string[]
  fulfillmentStatus?: string[]
  dateFrom?: Date
  dateTo?: Date
  search?: string
  page: number
  limit: number
}
```

Shared helper:

- `buildOrderListFilter()` in `repository-contract.js`.

## 11. Read Models

For alpha, repository queries can serve dashboard reads. Future read models are allowed when volume grows:

- `admin_order_list_view`
- `order_detail_view`
- `qr_performance_view`
- `product_availability_view`

Read models must remain workspace-scoped.

## 12. Soft Delete Rules

Repository must respect `deleted_at` for soft-deletable master data.

Default master-data query should exclude deleted rows where the table supports it.

Operational data should not be hard deleted:

- Orders.
- Payments.
- Audit logs.
- Payment events.

## 13. Repository Error Types

Repository errors are infrastructure/data-access errors. Services translate them to business errors when needed.

Canonical repository error examples:

- `DATABASE_CONNECTION_ERROR`
- `UNIQUE_CONSTRAINT_VIOLATION`
- `FOREIGN_KEY_VIOLATION`
- `RECORD_NOT_FOUND`
- `TRANSACTION_CONFLICT`
- `DEADLOCK_DETECTED`

Shared mapper:

- `normalizeRepositoryError(error, context)`.

## 14. Anti-Patterns

Avoid:

- Repository memanggil service lain.
- Repository memverifikasi payment webhook.
- Repository menghitung business status transition.
- Repository mengirim notification.
- Repository membaca data lintas workspace tanpa filter.
- Service langsung query database untuk domain persistence saat repository sudah tersedia.
- Controller langsung akses repository/database.
- Repository mengembalikan data mentah ke response public.
