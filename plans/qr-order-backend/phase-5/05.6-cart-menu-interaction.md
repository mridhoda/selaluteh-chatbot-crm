# Cart State Model

Wave 3 changes keep cart state as customer intent only. The new model helper is `web/src/features/public-store/utils/cartIntentModel.js`.

Local cart intent items contain only:

- `clientLineId`
- `productId`
- `quantity`
- `selectedModifierOptionIds`

Intent context contains only:

- `storefrontSlug`
- `outletId` when frontend outlet selection is allowed
- `qrSessionToken` when a QR session exists

The cart state does not store customer data, payment data, fulfillment status, payment status, final totals, provider payloads, or backend-owned authority fields.

# Validation Flow

`useGuestCart()` now builds a local preview cart for UI display and calls `phase5ApiClient.public.validateCart()` before checkout.

Validation request payload is produced by `buildCartValidationPayload()` and sends intent fields only. Backend response is normalized through `normalizeValidatedCart()` and tagged as `totalsAuthority: backend_validated`.

Local cart totals are produced only for preview and tagged as `totalsAuthority: local_preview_only`. Checkout never sends local totals.

# Invalidation Rules

The validated backend cart is cleared when:

- An item is added.
- An item is removed.
- Quantity changes.
- The cart is cleared.
- Outlet changes clear cart in the existing storefront and QR flows.

Backend product/modifier/outlet/QR errors are surfaced through validation state and safe error text. Browser-level inline item error rendering is limited until a component test runner and final backend error payload shape are available.

# UI Behavior

The existing mobile cart drawer remains the public cart UI. It displays local subtotal/fee/total as estimate only. Checkout page adds explicit backend validation CTA, backend-validated total display when available, and a warning that final total authority belongs to the backend.

For QR flows, locked outlet/location sessions pass only the QR session token to cart/checkout context. Selected `outletId` is included only when the frontend is allowed to select an outlet.
