# 05.9 Admin Order Management UI

## Scope

Wave 4 prepared Phase 5 admin order operations for the authenticated dashboard shell.

Update 2026-07-08: by user request, the active `/app/orders` route was restored to the legacy `OrdersPage` UI. `Phase5AdminOrdersPage.jsx`, the Phase 5 model/API safety work, and tests remain in the repository as reference/non-active integration assets. Phase 5 admin order requirements must now be integrated into the existing Orders UI/components rather than replacing the page.

The active route continues to use the current admin layout, sidebar, topbar, auth redirect behavior, workspace header behavior, and `ProtectedPage` permission guard for `orders:read`.

## Backend Contract

Real backend admin order routes were found in `server/src/routes/admin-orders.js`:

- `GET /api/v1/admin/orders`
- `GET /api/v1/admin/orders/:orderId`
- `POST /api/v1/admin/orders/:orderId/accept`
- `POST /api/v1/admin/orders/:orderId/prepare`
- `POST /api/v1/admin/orders/:orderId/ready`
- `POST /api/v1/admin/orders/:orderId/complete`
- `POST /api/v1/admin/orders/:orderId/cancel`

The frontend uses the Wave 1 `phase5ApiClient` admin methods and does not call generic status mutation endpoints.

## UI Behavior

The non-active Phase 5 admin page prototype shows:

- Order list from backend.
- Channel.
- Payment status.
- Fulfillment status.
- Outlet.
- QR/table context when returned.
- Backend total amount as display-only.
- Channel filter on loaded backend data.
- Outlet ID filter sent to backend as `outletId`.
- Detail panel with customer summary, item snapshot, payment status, fulfillment status, public order status, QR context, and status history.

The page strips secret-shaped and raw provider payload fields during normalization before rendering.

## Allowed Actions

Actions are rendered only from backend `allowed_actions`.

Supported backend action names:

- `accept_order`
- `mark_preparing`
- `mark_ready`
- `mark_completed`
- `cancel_order`

The frontend does not infer actions from `payment_status`, `fulfillment_status`, `public_order_status`, or local UI state.

## Explicit Endpoints

Action mapping:

- `accept_order` calls `acceptOrder`, which posts to `/api/v1/admin/orders/:orderId/accept`.
- `mark_preparing` calls `prepareOrder`, which posts to `/api/v1/admin/orders/:orderId/prepare`.
- `mark_ready` calls `readyOrder`, which posts to `/api/v1/admin/orders/:orderId/ready`.
- `mark_completed` calls `completeOrder`, which posts to `/api/v1/admin/orders/:orderId/complete`.
- `cancel_order` calls `cancelOrder`, which posts to `/api/v1/admin/orders/:orderId/cancel` with `{ reason }`.

Cancel requires a non-empty reason in the API client and UI model validation.

Duplicate submissions are blocked by a single in-flight action guard that disables all lifecycle buttons while a mutation is running.

After action success, the page updates from the returned order and refreshes detail to get updated `allowed_actions` where possible.

## Error Handling

Mapped admin lifecycle errors:

- `ORDER_INVALID_TRANSITION`
- `ORDER_UNPAID`
- `FORBIDDEN`

Errors are displayed as safe UI messages without raw stack traces or provider payloads.

## Regression Notes

The implementation intentionally avoids public checkout, public payment pending, public order tracking, WhatsApp webhook/order creation, CRM chat pages, and product/payment list rewrites.

Existing dashboard shell files were restored so `/app/orders` loads the legacy `OrdersPage` again and keeps the protected route in place.

The Phase 5 replacement page remains in the repo but is no longer the active `/app/orders` route. Existing WhatsApp order visibility is preserved through the legacy UI. Future Phase 5 work must add backend `allowed_actions`, explicit lifecycle endpoint usage, and safety/error handling into legacy Orders UI components without replacing the user-approved design.

## Tests

Added Node-testable model/API tests in `web/test/phase-5/admin-order-management.test.mjs`:

- List/detail normalization.
- Secret/raw provider payload stripping.
- Allowed actions rendering only from backend `allowed_actions`.
- No guessed actions from status.
- Explicit accept/prepare/ready/complete/cancel endpoints.
- No generic `PATCH` status endpoint.
- Cancel reason validation.
- Duplicate action prevention model validation.
- `ORDER_INVALID_TRANSITION`, `ORDER_UNPAID`, and `FORBIDDEN` safe error messages.

Existing Wave 1 API safety test was strengthened by making `phase5ApiClient.admin.cancelOrder` reject missing reason.
