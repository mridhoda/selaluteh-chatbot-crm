# 05.2 Frontend Architecture and App Shell

Tasks: `3.1` through `3.6`

Date: 2026-07-08

Scope: Architecture documentation only. Public UI pages and admin UI screens are intentionally not implemented in Wave 1.

## Audit

Current frontend evidence:

- Router: `react-router-dom` routes are defined through `web/src/routes/privateRoutes.jsx` and re-exported by app router files.
- Existing public routes: `/store/:storefrontSlug`, `/store/:storefrontSlug/checkout`, `/store/payment/pending/:checkoutToken`, `/store/order/:publicOrderToken`.
- Existing public shell: `web/src/features/public-store/layouts/PublicStoreLayout.jsx` with mobile-first public store components, but current data flow is mock/local-first.
- Existing admin shell: `/app/*` renders `DashboardLayout` and nested dashboard routes from `DashboardPage.jsx`.
- Existing admin orders screen: `/app/orders` uses legacy/mixed order behavior and must not become the Phase 5 lifecycle authority until explicit endpoint tests are in place.
- Current state approach: local React state plus localStorage/sessionStorage; no Redux/Zustand/React Query was found.
- Current test runner: Node built-in runner; browser/component/E2E runner is not configured.

Backend contract evidence:

- Public API is mounted at `/api/public` and `/api/v1/public`; Phase 5 client uses `/api/v1/public`.
- Admin Phase 5 order API is mounted at `/api/v1/admin/orders`.
- Admin QR management backend contract was not found.
- Payment gateway config route exists at `/payments/gateway/config`, but provider secrets and webhooks remain backend-only.

## Map

Architecture decisions:

- Routing: extend existing router conventions.
- Public shell: extend existing public store shell later, after replacing mock-only API usage.
- Admin shell: reuse existing admin shell and navigation patterns, but isolate Phase 5 order API methods from legacy generic status patching.
- State management: continue local feature state for Wave 2+; do not introduce a global store in Wave 1.
- Form validation: client validation is UX-only; backend remains final authority for product availability, modifiers, totals, payment status, fulfillment status, QR locks, and admin transitions.
- Environment/feature flags: frontend may use public feature flags only; no provider secrets or webhook secrets in frontend env.

## Route Architecture

Public route strategy:

- `/store/:storefrontSlug`: public online store storefront and menu.
- `/qr/:qrToken`: QR resolve entry route for Wave 2. This avoids overloading storefront routes and keeps path QR token as the only token needed for initial resolution.
- `/store/:storefrontSlug/checkout`: checkout form and submission after backend cart validation.
- `/store/payment/pending/:paymentId`: read-only payment pending/status route. The route must poll backend payment status and must not infer paid state from redirect or URL.
- `/store/order/:publicOrderToken`: public order tracking by opaque public token only.

Admin route strategy:

- `/app/orders`: current admin order route can become the Phase 5 order operations surface only after explicit endpoint and `allowed_actions` rendering tests exist.
- `/app/kitchen`: can reuse backend lifecycle actions for kitchen view later, but must use the same explicit Phase 5 API boundary.
- `/app/settings/payments` or existing `/app/payments`: payment gateway status/settings display may be integrated only with public-safe DTOs.
- Admin QR route remains blocked/hidden until backend admin QR contract exists.

Route compatibility rules:

- Do not remove existing CRM/chat/WhatsApp, products, platforms, payments, or legacy order routes in this wave.
- Do not wrap or rewrite the admin auth guard in Wave 1 because current login/session behavior is a high-regression area.
- Do not expose admin navigation, auth state, workspace IDs, raw provider payloads, or internal order IDs in public routes.

## Public Shell

Public shell goals for Wave 2+:

- Mobile-first layout using the existing public store visual pattern where safe.
- Header shows storefront/brand, selected outlet or QR locked context, and cart access.
- QR context display shows backend-provided outlet/location/table label only; it must not allow override for locked outlet/location QR modes.
- Cart access pattern should show local cart intent plus backend validation state separately.
- Checkout shell must present backend-validated totals as server response data, not client authority.
- Payment pending shell must show read-only status, retry/poll states, and safe fallback messages.
- Order tracking shell must show public-safe order number/token, queue/status, masked customer data, outlet/location label, and backend status timeline only.

Public shell safety rules:

- Public routes do not require login.
- Public routes must not use admin axios auth/workspace headers.
- Public routes must not call webhook/provider callback endpoints.
- Public routes must not persist sensitive customer/payment details beyond the minimum checkout UX need.

## Admin Shell Integration

Admin shell goals for Wave 5+:

- Reuse `DashboardLayout`, sidebar/topbar, page header, table, filters, detail drawer, status badge, confirm dialog, and toast patterns where safe.
- Order list/detail must read `allowed_actions` from backend response and render action buttons from that response only.
- Lifecycle buttons must call explicit endpoints: `accept`, `prepare`, `ready`, `complete`, and `cancel`.
- Cancel reason validation is UX-only; backend validates transition and reason requirements.
- Existing generic `PATCH /orders/:id/status` usage is legacy and must not be used for Phase 5 admin lifecycle actions.

Admin regression risks:

- `/app/chats` and WhatsApp commerce flows may depend on legacy order/payment behavior.
- `/app/products` and `/app/payments` contain broad direct API usage.
- Top-level admin auth and workspace behavior is currently spread across `httpClient`, `DashboardPage`, and storage.

## State Management

State model for Wave 2+:

- Storefront state: backend storefront/menu/outlet response, loading, error, selected outlet when backend allows selection.
- Cart intent state: product ID, quantity, modifier option IDs, client line ID, and selected outlet only when allowed.
- Backend-validated cart state: server-normalized items, totals, validation errors, unavailable items, modifier errors, and QR/outlet mismatch errors.
- QR session state: path QR token for resolve, backend QR session token only when backend returns/requires it, locked outlet/location display values.
- Checkout attempt state: `Idempotency-Key` generated per attempt, reused for retries of same attempt, replaced only for a new intentional checkout attempt.
- Payment polling state: read-only backend status, polling retry metadata, expiration/failure/manual review messages.
- Admin action state: per-order/per-action pending/error state; action availability still comes from backend `allowed_actions`.

Storage rules:

- localStorage/sessionStorage may store non-sensitive cart intent and UI preferences.
- Do not store provider secrets, webhook URLs as authority, raw provider payloads, payment credentials, or backend-owned final state.
- Customer contact data should be transient unless a later task adds explicit consent and retention requirements.

## Form Validation

Client-side validation is limited to UX feedback:

- Product/modifier UX: required modifier groups, quantity bounds, and obvious missing selections before cart validation.
- Customer info: required name/phone and basic phone shape before checkout submission.
- Checkout: submit disabled while pending and blocked until an `Idempotency-Key` exists.
- Cancel reason: require a non-empty reason when the action UI asks for one; backend remains final transition authority.
- Backend validation display: render backend error codes/messages through centralized mapping and item-level errors where available.

Authority rules:

- Frontend must not send `payment_status`, `fulfillment_status`, `public_order_status`, `allowed_actions`, or final totals as authority.
- Frontend must not override QR locked outlet/location.
- Frontend must not mark payment as paid.
- Frontend must not guess admin actions.

## Environment and Feature Flags

Allowed frontend env/config:

- Public API base URL from existing `VITE_API_BASE` or `VITE_API_URL`.
- Public feature flags for enabling hidden/incomplete routes after contracts exist.
- Sandbox display labels returned by backend-safe admin payment config.

Disallowed frontend env/config:

- Provider API keys, webhook secrets, signing secrets, merchant secrets, or private callback credentials.
- Any flag that lets frontend bypass idempotency, QR lock, backend cart validation, payment status, or admin transition authority.

## Risks and Tests

Primary risks:

- Existing public store UI is mock/local-first and may leak local total assumptions if reused without refactor.
- Admin order UI currently has legacy generic status patch behavior.
- Admin QR management backend contract is missing.
- Browser/component/E2E runner is not configured.
- Shell/process policy currently blocks local command verification.

Required tests before UI implementation:

- API client safety tests for forbidden payload fields, idempotency, webhook prevention, explicit admin endpoints, and error mapping.
- Public storefront/QR route tests for loading, invalid QR/storefront, locked outlet/location display, and no admin auth exposure.
- Cart/checkout tests for backend validation, duplicate checkout prevention, and no final total/status authority in requests.
- Payment/public order tests for read-only status and public data privacy.
- Admin order tests for backend `allowed_actions`, explicit endpoint calls, pending/error states, and no generic PATCH status.

## Verification

Command verification remains blocked by the workspace shell/process policy documented in prior Wave 0 artifacts. Wave 1 code changes are covered by Node test files that should run with:

```bash
npm --prefix web test -- --test-name-pattern "Phase 5|checkout|admin lifecycle|backend error"
npm --prefix web test
npm --prefix web run lint
npm --prefix web run build
```

## Known Gaps

- Admin QR route is documented but not implementable until backend admin QR contract exists.
- Public UI pages are intentionally deferred to Wave 2+.
- Admin UI lifecycle rendering is intentionally deferred to Wave 5+.
- Exact payment settings editing scope is not implemented; only safe read boundary is documented for future work.
