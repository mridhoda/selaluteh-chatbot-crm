## Map

Framework, router, bundler, state, tests, styling, API client:

- Framework: React `^19.2.7` with `react-dom` `^19.2.7`.
- Router: `react-router-dom` `^6.26.1`; `BrowserRouter` is created in `web/src/app/App.jsx` with `future={{ v7_startTransition: true, v7_relativeSplatPath: true }}`.
- Route implementation: top-level routes live in `web/src/routes/privateRoutes.jsx`. `web/src/app/router.jsx`, `web/src/app/routes.jsx`, and `web/src/routes/publicRoutes.jsx` re-export existing route modules.
- Bundler/dev server: Vite `^8.0.16` with `@vitejs/plugin-react` and `@tailwindcss/vite` in `web/vite.config.js`.
- State management: local React state/hooks plus `localStorage` and `sessionStorage`; no Redux/Zustand/React Query store found. `auth.store.js` is a one-line placeholder. Providers only wrap `ToastProvider` and start push/realtime services when a token exists.
- Test runner: Node built-in test runner via `node --test "test/**/*.test.mjs"`.
- Styling: global CSS files, module CSS files, inline styles, Tailwind utility classes through Tailwind/Vite, CSS custom properties, FontAwesome, Lucide, Chart.js, and Motion dependency. No single strict design-system package was found.
- Admin/private API client: centralized axios instance in `web/src/shared/api/httpClient.js`; base URL from `VITE_API_BASE`, `VITE_API_URL`, dev origin, host `:5000`, or `http://localhost:5000`; request interceptor attaches `Authorization: Bearer <token>` and `x-workspace-id`; response interceptor redirects to `/login` on 401 when a token existed unless `skipAuthRedirect` is set.
- Public store API client: `web/src/features/public-store/api/publicStoreApi.js` is currently hardwired to `USE_MOCK_PUBLIC_STORE = true` and returns local mock data. `publicStoreEndpoints.js` documents intended endpoint strings but is not used by `publicStoreApi.js`.

Public routes:

- `/` landing page.
- `/login`, `/register`, `/verify`, `/forgot-password`, `/reset-password/:token` auth pages.
- `/store/:storefrontSlug` public storefront.
- `/store/:storefrontSlug/checkout` public checkout.
- `/store/payment/pending/:checkoutToken` public payment pending page.
- `/store/order/:publicOrderToken` public order status page.
- `*` redirects to `/`.

Admin routes under `/app/*`:

- `/app` dashboard overview.
- `/app/chats`, `/app/analytics`, `/app/contacts`, `/app/complaints`, `/app/escalation-inbox`, `/app/escalation-settings`.
- `/app/orders`, `/app/kitchen`, `/app/products`, `/app/outlets`, `/app/payments`.
- `/app/platforms`, `/app/agents`, `/app/agents/:id`, `/app/agents/:id/:tab`.
- `/app/humans`, `/app/human-agents`.
- `/app/reports`, `/app/access-control`, `/app/settings`, `/app/billing`, `/app/profile`.

Auth guards and layouts:

- `web/src/routes/routeGuards.jsx` exports `RequireAuth`, but it only returns `children`; it does not enforce authentication.
- Top-level `/app/*` route renders `DashboardLayout`, which re-exports `DashboardPage`; no top-level auth guard wraps `/app/*` in `privateRoutes.jsx`.
- `DashboardPage` redirects to `/login` only when its local `useAuth()` has no `user` parsed from `sessionStorage`. It does not read `localStorage` there, while the API client reads both stores for tokens.
- `ProtectedPage` inside `DashboardPage` applies permission checks for nested admin pages using `hasPermission()` and `isOwner()`.
- Sidebar navigation visibility uses `canAccessNavItem()` and the local/frontend permission matrix plus optional workspace access data fetched from `/api/workspaces/current/access`.
- `AppLayout.jsx` and `AuthLayout.jsx` are pass-through wrappers. `DashboardLayout.jsx` re-exports the dashboard page. Public store uses `PublicStoreLayout.jsx`.
- Logout in `Topbar.jsx` removes tokens/user from session/local storage and navigates to `/login`.

Environment variables observed:

- From `web/.env.example`: `VITE_API_BASE`, `VITE_API_URL`, `VITE_APP_NAME`, `VITE_DEMO_MODE`, `VITE_CLEANTER_ENABLED`, `VITE_CLEANTER_BASE_URL`, `VITE_CLEANTER_PRINT_PATH`, `VITE_CLEANTER_TIMEOUT_MS`, `VITE_CLEANTER_MAX_PAYLOAD_BYTES`, `VITE_CLEANTER_SUPPORTS_CUT`.
- Additional referenced variable found in code: `VITE_PUBLIC_FILES_BASE_URL` in `AgentDetail.jsx`.
- Dev proxy variable in `vite.config.js`: `VITE_DEV_API_PROXY_TARGET`, defaulting to `http://127.0.0.1:5000`.

Current public-store behavior:

- Storefront loads mocked storefront/categories/products through `usePublicStorefront()` and `publicStoreApi.getStorefront()`.
- Storefront allows outlet selection and persists it to `localStorage` under `public-store-outlet:<storefrontSlug>`.
- Guest cart is local state/localStorage based and calculates display totals client-side for preview.
- Checkout uses local/mock customer form validation and calls mocked `publicStoreApi.checkout()` without a real backend request and without an `Idempotency-Key` header.
- Checkout contains mock public customer login/register UI storing `public-store:user` in `localStorage`; this is not connected to the real auth API.
- Payment pending page reads mocked payment status, opens `paymentUrl` in a new tab, shows a local countdown, and does not mark payment paid.
- Public order status page reads mocked order data and displays success-style copy, masked phone, outlet, item summary, invoice actions, and a refresh button.

Current admin order/product/payment behavior:

- Orders page fetches `/orders`, `/outlets`, and per-agent `/agents/:id`; also includes large mock order fixtures merged into the page baseline. It has filtering, pagination, export, detail drawer, realtime event listeners, status patching through `PATCH /orders/:orderId/status`, cancellation through `PUT /orders/:orderId/cancel`, delete through `DELETE /orders/:orderId`, and a mock/demo override path.
- Products page is a large mixed implementation with dummy product data plus backend calls to `/products`, `/outlets`, `/orders`, `/agents/:id`, `/products/:id/outlet-availability`, `/api/inventory/:id`, `/api/inventory/:id/adjust`, `/audit/logs`, `/files/upload`, and product create/update/delete/archive operations.
- Payments page uses `paymentsApi` for `GET /payments`, `GET /payments/:id`, `GET /payments/:id/events`, `POST /payments/:id/resend-link`, and `POST /orders/:orderId/payment-link`. It maps backend fields into a table/detail UI with local filtering, provider/method/reconciliation summaries, copy/open payment link actions, and order/chat navigation hooks.
- Kitchen page and order admin routes use frontend permission checks for `orders.manage_status`, but admin allowed actions are not yet confirmed as backend-driven in the audited code.

Current CRM/chat/marketplace behavior:

- Chat center/dashboard inbox fetches `/chats`, `/agents`, and `/platforms`, polls every 4 seconds, refreshes on window focus, supports search/filter tabs, send message through `/chats/:id/send`, takeover/release through `/chats/:id/takeover` or `/chats/:id/release`, resolve/open via `PUT /chats/:id`, and delete through `DELETE /chats/:id`.
- Chat context/cart panel calls cart and checkout/order/payment endpoints including `/carts`, `/carts/:id/items`, `/checkouts`, `/checkouts/:id/confirm`, `/orders`, `/orders/:id/payments/session`, `/payments`, `/api/workspaces/settings/payment`, and `/carts/clear-active`.
- Platforms page/API area manages connected platforms with `/platforms` and related agent/platform dependencies. This is the closest current "marketplace/connected platforms" baseline found; no separate marketplace module was found in the audited module list.
- Agents, contacts, complaints, escalation, reports, billing, settings, outlets, access control, and profile are integrated under the admin shell and share the same axios client/session behavior.

Backend contract and error shape baseline:

- Admin/private modules call many direct endpoint strings through the centralized axios client. Response shapes vary by module: some expect arrays directly, some expect `{ data: [...] }`, and some expect nested `{ data: { ... } }`.
- `getApiErrorMessage()` only reads `error.response.data.error`, then `error.message`, then fallback.
- Public store intended endpoints are present in `publicStoreEndpoints.js`, but the current public store implementation does not consume them.

Classification for future Phase 5 work:

- Public store/QR/checkout/payment/order tracking: rebuild/greenfield against real Phase 4 contract because current code is mock-first and local-state-first.
- Shared API base/client: reuse with extension, because a centralized axios client exists for authenticated APIs, but public no-login APIs need explicit contract-safe calls and headers such as `Idempotency-Key`.
- Admin shell/navigation: reuse with targeted refactor, because routes/layout/sidebar exist but auth guard and backend-driven admin allowed-actions are incomplete.
- Admin order/payment/product pages: extend/refactor only where Phase 5 requires contract consumption; avoid broad rewrites during later task slices.
- Existing CRM/chat/WhatsApp flows: regression-check, not rebuild, unless a task directly touches shared routing, API client, auth, order, product, payment, marketplace, CRM, or chat surfaces.
