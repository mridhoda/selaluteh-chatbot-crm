## API, Auth, and State Audit

API client boundaries:

- Authenticated/admin client: `web/src/shared/api/httpClient.js` exports an axios instance.
- Base URL: `web/src/shared/api/apiBase.js` reads `VITE_API_BASE` or `VITE_API_URL`, then falls back to dev same-origin or host `:5000`.
- Error helper: `web/src/shared/api/apiError.js` currently maps `error.response.data.error`, then `error.message`, then fallback.
- Public store API: `web/src/features/public-store/api/publicStoreApi.js` is currently hardwired to `USE_MOCK_PUBLIC_STORE = true` and does not call backend.
- Intended public endpoints: `web/src/features/public-store/api/publicStoreEndpoints.js` defines `/public/storefronts/:slug`, guest sessions, carts, checkout, payment status, and public order paths, but these differ from active backend route evidence under `server/src/routes/public-store.js`.

Direct API call patterns:

- Most admin calls use the shared axios instance, but endpoint strings are spread across page/component files.
- Direct endpoint usage is extensive in `DashboardPage.jsx`, `ProductsPage.jsx`, `OrdersPage.jsx`, `ChatContextPanel.jsx`, `Sidebar.jsx`, and module-specific APIs.
- Existing admin orders page uses legacy endpoints including `PATCH /orders/:orderId/status`; Phase 5 admin order UI must prefer explicit backend action endpoints from `server/src/routes/admin-orders.js`.

Auth/session:

- `httpClient.js` attaches `Authorization: Bearer <token>` from localStorage/sessionStorage.
- `httpClient.js` attaches `x-workspace-id` from stored user/workspace data.
- 401 responses clear token/session and redirect to `/login` only when a token existed and `skipAuthRedirect` is not set.
- Frontend permissions in `web/src/shared/auth/permissions.js` are UI gates only, not backend authority.

State management:

- No Redux/Zustand/React Query store was found.
- State is local React state/hooks plus localStorage/sessionStorage persistence.
- Public store cart/outlet selection currently persists in localStorage.
- Public checkout user mock stores `public-store:user` in localStorage and is not real backend auth.

## Backend Contract Audit

Confirmed backend route/service evidence:

- Public storefront: `GET /stores/:storefrontSlug` and `GET /storefronts/:storefrontSlug` in `server/src/routes/public-store.js` call `getPublicStorefront`.
- QR resolve: `GET /qr/:qrToken` calls `getQrStoreContext` for v1 public paths or `getQrContext` for legacy public paths.
- Cart validation: `POST /carts/validate` calls `validatePublicCart` and strips internal context before response.
- Checkout: `POST /checkout` calls `createPublicCheckout` with `Idempotency-Key` from header/body.
- Payment status: `GET /payments/:paymentId/status` calls `getPublicPaymentStatus`.
- Public order tracking: `GET /orders/:publicOrderToken` calls `getPublicOrderByToken`.
- Admin Phase 5 orders: `server/src/routes/admin-orders.js` has list/detail plus explicit `accept`, `prepare`, `ready`, `complete`, and `cancel` endpoints and maps `allowed_actions` from backend capabilities/permissions.
- Legacy admin orders: `server/src/routes/orders.js` still exposes generic `PATCH /orders/:id/status`; this is a regression risk for Phase 5 admin UI.
- Admin payment settings/status: `server/src/routes/payments.js` exposes `/payments/gateway/config` and `workspace-settings` routes exist, but secret handling must remain backend-only.
- Admin QR settings: no `server/src/routes/*qr*` route file was found. QR runtime service exists, but admin QR management contract is missing for frontend implementation.

Backend authority rules confirmed:

- Backend computes public cart totals and checkout/order/payment state.
- Backend requires checkout idempotency key.
- Backend derives public order status from payment/fulfillment.
- Backend admin order mapping returns `allowed_actions`.
- Frontend must not call payment webhook routes or expose provider payloads.

Contract mismatches/gaps:

- Frontend `publicStoreEndpoints.js` endpoint strings do not exactly match active route evidence; later API client work must reconcile mount prefixes and path names.
- Error code naming differs in docs/tasks versus runtime for idempotency required: task mentions `CHECKOUT_IDEMPOTENCY_REQUIRED`, runtime currently throws `IDEMPOTENCY_KEY_REQUIRED` in `createPublicCheckout`.
- Admin QR settings contract is missing.
- Universal QR support is documented as incomplete in prior Phase 4 evidence.

## Risks, Missing Contracts, and Blockers

Risks:

- Public store runtime is mock-backed and local-state-first.
- Public checkout currently lacks real backend `Idempotency-Key` integration in frontend.
- Existing admin orders page can use generic status patching and mixed mock/demo data.
- Frontend permission matrix may be mistaken for backend allowed-action authority if not isolated.
- Public order tracking displays mocked order status and must avoid internal IDs/provider payloads when integrated.
- Endpoint mount prefixes are not fully reconciled between frontend endpoint constants and Express route mounting.

Missing/blocked contracts:

- Admin QR settings/list/detail/action contract is not found in backend routes.
- Browser/component/E2E test runner is not present.
- Phase 5 `spec.yaml` and `design.md` are missing.
- Verification commands are blocked by workspace shell/process policy.

Open questions for later waves:

- Confirm final mounted public API prefix for frontend: `/public/*`, `/api/v1/public/*`, or another prefix.
- Decide whether public store existing mock UI is refactored in place or replaced route-by-route.
- Confirm admin QR scope for alpha or keep hidden until backend route exists.
- Confirm payment settings UI contract for secret-safe provider configuration edits versus read-only status.

## Audit Conclusion

Phase 5 should proceed with an API-client-first integration path. Existing UI can be reused visually in places, but public order/checkout/payment authority and admin order actions require contract-safe rebuild/refactor behind tests.
