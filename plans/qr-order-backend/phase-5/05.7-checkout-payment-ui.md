# Checkout Form

`useCheckoutForm()` now submits through the centralized Wave 1 `phase5ApiClient`. The form collects customer name, normalized phone, and optional note. These values are not persisted in cart storage.

Checkout requires a backend-validated cart. If no valid backend cart is present, the hook calls `validateCart()` before mutation. The checkout payload is generated by `buildCheckoutPayload()` and includes only:

- Storefront/QR/outlet intent context.
- Customer contact fields needed for order creation.
- Cart intent items.

The payload does not include final totals, `payment_status`, or `fulfillment_status`.

# Idempotency-Key Lifecycle

Checkout uses `createCheckoutAttempt()` to generate one `Idempotency-Key` per active checkout attempt. The same key is reused when retrying the same attempt after a safe failure. The checkout page exposes a "Percobaan Baru" action to intentionally start a new attempt.

Duplicate mutation prevention is enforced by:

- Returning early when `submitting` is already true.
- Disabling submit while pending.
- Requiring backend-validated cart before mutation.
- Keeping idempotency key in hook state for retry safety.

`IDEMPOTENCY_CONFLICT` maps to a safe user-facing error.

# Payment Pending Flow

Checkout navigates to `/store/payment/pending/:paymentId` after backend order/payment creation. The payment pending page reads status through `phase5ApiClient.public.getPaymentStatus()` only.

Payment URL is displayed/opened only as a payment action. It is never treated as a paid signal. Paid/failed/expired/manual-review states come only from backend status response.

# Polling Strategy

`usePaymentStatus()` polls the read-only payment status endpoint with a bounded interval and max poll count. Polling stops when terminal statuses are observed:

- `paid`
- `failed`
- `expired`
- `cancelled`/`canceled`
- `manual_review`

The hook normalizes status with `normalizePaymentStatus()` and does not expose raw provider payload.

# Safety Rules

Checkout and payment UI follows these authority rules:

- Backend cart validation supplies final totals.
- Local subtotal is an estimate only.
- Checkout includes QR session token when present.
- Checkout includes selected outlet only when allowed.
- Checkout never sends payment/fulfillment status.
- Redirect/payment URL never marks an order paid.
- Payment pending reads status only and hides raw provider payload.

# Tests

Coverage added in `web/test/phase-5/cart-checkout-payment-order-model.test.mjs` and existing `phase5-api-client-safety.test.mjs`:

- Checkout payload omits forbidden fields and totals authority.
- API client requires and sends `Idempotency-Key`.
- Idempotency retry reuses the same key and intentional new attempt generates a new key.
- Duplicate checkout mutation guard logic.
- Redirect/payment URL is not paid.
- Polling terminal behavior.

# Limitations

Provider error UI is covered by safe hook error handling and API error mapping, but not by a DOM assertion because the project currently has no browser/component runner. Full payment pending polling behavior still requires sandbox/manual verification in a later stabilization wave.
