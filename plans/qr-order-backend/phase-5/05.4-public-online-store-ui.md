# 05.4 Public Online Store UI

Tasks: `5.1` through `5.6`

Date: 2026-07-08

Scope: Public `/store/:storefrontSlug` storefront route, backend storefront loading through the Phase 5 API boundary, outlet/product/modifier UI adaptation, Node-testable UI model tests, and safety notes. Checkout/cart validation/payment/order tracking are not implemented in this wave.

## Audit

Existing frontend evidence:

- `/store/:storefrontSlug` already existed in `web/src/routes/privateRoutes.jsx`, but it is a top-level route outside `/app/*` and therefore does not use `DashboardLayout` or admin auth shell.
- Existing public store UI in `web/src/features/public-store/pages/StorefrontPage.jsx` was mobile-first and reusable, but its hook loaded mock/local data through `publicStoreApi`.
- Existing `ProductModifierSheet` displayed a local preview total with text that final price is validated by backend.
- Existing `ProductCard` disabled unavailable products through `product.isAvailable`.
- No browser/component test runner is configured; the project test script uses Node built-in test runner only.

## Implementation

Files changed for public store UI:

- `web/src/features/public-store/hooks/usePublicStorefront.js`
- `web/src/features/public-store/pages/StorefrontPage.jsx`
- `web/src/features/public-store/components/OutletPickupBadge.jsx`
- `web/src/features/public-store/components/ProductModifierSheet.jsx`
- `web/src/features/public-store/utils/publicStoreModel.js`
- `web/test/phase-5/public-store-ui-model.test.mjs`

Route behavior:

- `/store/:storefrontSlug` remains a public no-login route outside `/app/*`.
- Storefront data now loads through `phase5ApiClient.public.getStorefront(storefrontSlug)`.
- The frontend normalizes backend response variants with `normalizeStorefrontResponse()` rather than relying on mock-only shape.
- Invalid or missing storefront responses render a safe public error state without admin shell or internal details.

Screens and states:

- Loading: existing `StoreSkeleton`.
- Error: `StoreErrorState` with safe mapped messages from `getSafePublicStoreError()`.
- Empty storefront: `StoreErrorState` with invalid link copy.
- Inactive store: `StoreErrorState` with online ordering unavailable copy.
- No active outlets: `StoreErrorState` after rendering safe storefront identity header.
- Empty products: existing `ProductGrid` empty state.

## API Dependencies

Public API dependency:

- `GET /api/v1/public/storefronts/:storefrontSlug` through `phase5ApiClient.public.getStorefront()`.

Accepted response fields are normalized from backend-provided data only:

- `storefront`
- `outlets` or `eligibleOutlets`
- `menu.categories` or `categories`
- `menu.products` or `products`
- Product availability fields such as `isAvailable` or `availability`
- Product price preview fields such as `basePriceMinor`, `pricePreviewMinor`, or `priceMinor`

Safety rules:

- Frontend displays backend-provided price preview only.
- Frontend does not compute final order total authority.
- Frontend does not send payment status, fulfillment status, final total, or order status in this wave.
- Local cart item preview remains UX-only and is not a backend authority.

## Component Decisions

Storefront route:

- Existing mobile-first components were preserved to reduce regression risk.
- `usePublicStorefront()` was switched from mock API to Wave 1 `phase5ApiClient`.
- `normalizeStorefrontResponse()` adapts backend responses into the existing component shape.

Outlet selection:

- Outlet options come from backend response only.
- Disabled/inactive outlets are filtered out of selectable outlets.
- Changing outlet clears local cart intent to avoid carrying stale outlet-specific item intent into later validation.

Product list:

- Products are grouped/filterable by backend categories through the existing category/search controls.
- Sold-out/unavailable products normalize to `isAvailable: false` and remain disabled by `ProductCard`.

Product detail/modifier picker:

- Product details render name, description, image fallback, and backend-provided price preview.
- Required and optional modifier groups are rendered from backend response.
- `validateModifierSelection()` is a UX-only guard for missing required modifiers or max-select violations.
- Add-item output remains cart intent: `productId`, `quantity`, `selectedModifierOptionIds`, and optional note.

## Tests

Added Node-testable model tests:

- `web/test/phase-5/public-store-ui-model.test.mjs`

Covered behavior:

- Storefront loading normalization with identity, outlets, categories, and products.
- Invalid storefront normalization to safe empty state.
- Outlet selector exposes only active backend-provided outlets.
- Sold-out product normalization prevents available state.
- Modifier picker validation for missing required modifiers.
- Safe invalid storefront/internal error mapping.
- Safe expired/revoked QR error mapping shared by public error system.

Documented test limitation:

- Browser component tests for actual DOM clicks, responsive layout, and modal controls were not added because no supported browser/component test runner exists in the project.
- The added tests cover the pure model and API-boundary behavior that the React components consume.

## Known Limitations

- Full component/browser coverage for mobile controls is documented but not complete.
- Local cart UI still exists only as intent/preview state; backend cart validation and checkout are Wave 3+.
- Existing mock `publicStoreApi` remains for older checkout/payment/order components not in this wave; `/store/:storefrontSlug` storefront loading no longer uses it.
