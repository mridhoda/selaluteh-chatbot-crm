# 05.12 Frontend Testing and E2E Stabilization

## Test Strategy

The project currently uses Node's built-in test runner through:

```sh
npm --prefix web test
```

Wave 5 keeps this convention and does not add unsupported browser/E2E dependencies.

The test strategy is split into:

- Node-testable API client safety tests.
- Node-testable public/admin model tests.
- Contract fixture tests using Wave 0 fixtures.
- E2E/manual flow plans documented as blocked until a browser runner or manual sandbox is available.

## Automated Coverage Added or Expanded

API client and auth/idempotency/QR safety:

- `web/test/phase-5/phase5-api-client-safety.test.mjs`
- Public checkout requires and sends `Idempotency-Key`.
- Checkout rejects backend-owned authority fields.
- Public methods use expected storefront, QR resolve, cart validation, payment status, and public order endpoints.
- Admin lifecycle methods use explicit endpoints and never generic status `PATCH`.
- Webhook/provider callback endpoints are blocked.
- Backend error mapping covers required Phase 5 codes.
- Raw stack/provider error text is suppressed.
- Request ID is preserved safely.
- Admin auth/workspace headers are sent.

Public cart/checkout/payment/order models:

- `web/test/phase-5/cart-checkout-payment-order-model.test.mjs`
- Cart intent stores only safe product/modifier/quantity fields.
- Local totals remain preview-only.
- Checkout payload omits totals and payment/fulfillment authority.
- Duplicate checkout guard requires non-submitting, items, and backend validation.
- Payment redirect URL is never treated as paid.
- Payment polling stops on terminal states and slows on rate limit.
- Public order model hides internal IDs, raw provider payload, admin notes, admin user data, and sensitive customer data.
- Stored cart intent excludes customer contact and authority fields.

Admin order models:

- `web/test/phase-5/admin-order-management.test.mjs`
- Admin order list/detail strips secrets, raw provider payloads, admin notes, and admin user data.
- Actions render only from backend `allowed_actions`.
- Action endpoints are explicit.
- Cancel requires reason.
- Duplicate admin action is rejected by model guard.
- Lifecycle errors map to safe messages.

Existing Wave 0-4 suites remain in place for:

- Env guard.
- Contract fixtures.
- Public store UI model.
- QR store UI model.
- Admin QR/payment settings model.
- Cart/checkout/payment/order model.
- Admin order model.

## E2E Flow Plans

No browser/E2E runner is configured in `web/package.json`, and no foreground command runner is available in this environment. The following E2E plans are therefore documented as blocked, not passed.

### Online Store Flow

Planned checks:

- Open public storefront `/store/:storefrontSlug`.
- Select outlet.
- Add product with modifiers.
- Validate cart with backend.
- Submit checkout with `Idempotency-Key`.
- Reach payment pending page.
- Read payment status from backend.
- Open public order tracking.
- Confirm payment URL/redirect does not mark order paid.
- Confirm no final total/payment/fulfillment authority is client-owned.

Blocked result: not executed due missing E2E/browser runner.

### QR Store Flows

Planned checks:

- Universal QR checkout success.
- Outlet QR checkout success with locked outlet.
- Location QR checkout success with locked outlet/location.
- Universal QR missing outlet shows safe error.
- Outlet mismatch shows safe error.
- Location mismatch shows safe error.
- Expired/revoked QR shows safe message without stack/provider payload.
- Frontend cannot override locked outlet/location in payload.

Blocked result: not executed due missing E2E/browser runner and backend sandbox session.

### Admin Order Lifecycle Flow

Planned checks:

- Open admin order list.
- Open order detail.
- Render backend `allowed_actions` only.
- Accept paid order.
- Mark preparing.
- Mark ready.
- Mark complete.
- Cancel requires reason.
- Unpaid order action is blocked/handled by backend safe error.
- Order refreshes after action.
- Duplicate clicks do not create duplicate submissions.

Blocked result: not executed due missing E2E/browser runner and authenticated staging session.

### Security and Regression Checks

Planned checks:

- Public UI does not show internal order ID.
- Public UI does not show raw provider payload.
- Public UI does not show admin notes/admin user data.

- Payment redirect is not treated as paid.
- Existing admin navigation still works.
- Existing WhatsApp order visibility still works.
- Existing CRM/chat screens still load.

Blocked result: not executed due missing E2E/browser runner. Model/API tests cover the non-DOM portions.

## Known Failures and Blockers

- No E2E/browser runner exists in `web/package.json`.
- No foreground shell runner is available in this environment for `npm` commands.
- No staging/backend/browser session is available for manual sandbox verification.
- Component DOM tests are represented by model/API tests only; React component assertions require a supported runner in a later wave.
- Admin QR management remains blocked by missing backend admin QR contract from Wave 4.

## Readiness Impact

Wave 5 automated model/API stabilization is implemented where feasible.

Wave 6 may proceed only as alpha-readiness review and QA planning, not as GO approval, until browser/E2E/manual sandbox blockers are resolved and the No-Go checklist is verified in a runnable environment.
