# 05.11 Frontend Security, Error, and State Handling

## Scope

Wave 5 hardens Phase 5 public/admin frontend behavior for safe display, duplicate mutation prevention, bounded polling, and privacy-oriented state handling.

The frontend remains an intent sender and contract consumer only. Backend remains the authority for price, totals, QR locks, payment status, fulfillment status, public order status, and admin allowed actions.

## Error Mapping

Shared error handling is centralized in `web/src/shared/api/apiError.js`.

Mapped backend codes include:

- `QR_EXPIRED`
- `QR_REVOKED`
- `QR_OUTLET_MISMATCH`
- `QR_LOCATION_MISMATCH`
- `PRODUCT_UNAVAILABLE`
- `MODIFIER_INVALID`
- `CHECKOUT_IDEMPOTENCY_REQUIRED`
- `IDEMPOTENCY_KEY_REQUIRED`
- `IDEMPOTENCY_CONFLICT`
- `PAYMENT_PROVIDER_ERROR`
- `ORDER_INVALID_TRANSITION`
- `ORDER_UNPAID`
- `FORBIDDEN`
- `RATE_LIMITED`
- `INTERNAL_ERROR`

`mapBackendError()` now sanitizes fallback backend messages so raw stack traces, provider payload references, webhook/signature/API-key/secret/token-shaped text, and raw provider payload labels do not become public/admin UI copy.

`getSafeErrorDisplay()` exposes:

- Safe message.
- Optional support request ID.
- Optional support text.
- Retry eligibility for safe retry UI.

## Loading and Empty States

Existing Phase 5 UI includes loading/empty states for the critical surfaces:

- Storefront and checkout use `StoreSkeleton` and `StoreErrorState`.
- QR Store uses safe QR error state and empty menu/outlet handling from Wave 2.
- Cart validation and checkout expose pending/invalid/error states from `useGuestCart()` and `useCheckoutForm()`.
- Payment pending exposes read-only status loading, manual refresh, and error feedback.
- Public order tracking uses `StoreSkeleton`, `StoreErrorState`, safe status cards, and refresh.
- Admin order list/detail expose loading, empty table, no selected order, no item snapshot, no status history, no allowed action, and action error states.
- Admin QR/payment settings expose read-only provider status and blocked QR contract state.

Browser/mobile visual verification remains blocked because no browser/E2E runner is available in this environment.

## Privacy Safeguards

Public order model sanitization is enforced by `sanitizePublicOrder()` in `cartIntentModel.js`:

- Allows only public order token/number, queue number, public status fields, masked customer phone, outlet, QR display context, items, totals, invoice links, and timestamps.
- Drops raw/internal identifiers such as `id` and `internalOrderId`.
- Drops raw provider payloads.
- Drops admin notes and admin user data.
- Drops full customer phone and email.
- Allows only same-origin or relative invoice links.

Cart storage is intent-only:

- `productId`
- `quantity`
- `selectedModifierOptionIds`
- `clientLineId`

It does not persist customer contact data, payment status, fulfillment status, final totals, provider payloads, or backend authority fields.

Admin order normalization strips secret/raw-provider/admin-private keys before rendering. Generic `notes` is no longer used as a customer-note fallback to avoid displaying internal admin notes accidentally.

Frontend env secret checks remain covered by `web/test/phase-5/env-guard.test.mjs` from Wave 0.

## Polling, Retry, and Concurrency

Checkout safeguards:

- Duplicate submit returns early while `submitting` is true.
- Submit button is disabled while pending.
- Checkout requires a backend-validated cart.
- Checkout requires `Idempotency-Key` through `phase5ApiClient.public.checkout()`.
- Retry of the same attempt reuses the same idempotency key.
- A new attempt requires explicit user action.
- The UI does not assume mutation success from local state; success navigation only happens after backend checkout response.

Admin action safeguards:

- `validateAdminAction()` rejects a second action while `inFlightAction` is set.
- All lifecycle buttons are disabled while any admin action is in flight.
- Cancel requires a reason in the UI model and API client.
- Actions are rendered only from backend `allowed_actions`.
- The UI refreshes order detail after successful action where possible.

Payment polling safeguards:

- `usePaymentStatus()` reads only the public payment status endpoint.
- Polling stops on `paid`, `failed`, `expired`, `cancelled`/`canceled`, and `manual_review`.
- Polling slows on `rate_limited`/`throttled` via `getNextPaymentPollingDelay()`.
- Payment URL/redirect is never treated as paid.

## Accessibility Notes

Implemented accessibility-oriented basics:

- Checkout key form fields have labels.
- Admin filters have `aria-label`.
- Destructive cancel action requires confirmation and reason.
- Disabled states are used for in-flight checkout/admin actions.

Limitations before alpha:

- No automated browser accessibility runner exists in the project.
- No mobile viewport screenshot pass was runnable in this environment.
- Full keyboard/focus regression requires browser/manual verification.
