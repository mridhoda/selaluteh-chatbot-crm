# 05.10 Admin QR and Payment Settings UI

## Scope

Wave 4 adds a safe admin status page at `/app/payments/settings` using the existing dashboard shell and `payments:read` guard.

This page intentionally limits authority to backend-returned read-only data.

## Admin QR Management

Admin QR management is blocked by backend contract.

Audit result:

- Public QR resolve exists at `GET /api/v1/public/qr/:qrToken`.
- No backend admin QR list/detail/create/update/activate/disable/revoke route was found under `server/src/routes`.

Implemented UI:

- Read-only blocked status card.
- Explanation that frontend will not invent QR authority.
- Deferred capability list for list/detail/create/activate/disable/revoke/print-export.

Not implemented because backend contract is missing:

- QR list/detail retrieval.
- Universal QR creation.
- Outlet QR creation.
- Location QR creation.
- Activate/disable/revoke actions.
- Printable QR generation/export.

## Payment Provider Settings

Backend read-only route exists at `GET /payments/gateway/config`.

The admin settings page calls `phase5ApiClient.admin.getPaymentGatewayConfig()` and displays:

- Active provider.
- BayarGG display label when provider is `bayargg`.
- Environment/mode if returned.
- Configured status.
- Backend-managed webhook route.
- Public base URL.

The page does not provide provider configuration, activation, or disable actions because no safe backend management endpoint for those actions was included in the audited Phase 5 frontend contract.

## Security Rules

The UI and model layer strip or avoid rendering keys matching secret/raw-provider patterns, including:

- API keys.
- Webhook secrets.
- Tokens.
- Signatures.
- Raw provider payloads.

The frontend does not call BayarGG, Xendit, Doku, Midtrans, or any provider API directly. It calls only the backend gateway config route.

## Tests

Added Node-testable tests in `web/test/phase-5/admin-qr-payment-settings.test.mjs`:

- QR admin management blocked status when backend contract is missing.
- Deferred QR capabilities are documented by model output.
- Payment gateway config normalization.
- Secrets and raw provider payload values are stripped.
- Payment settings client calls backend `/payments/gateway/config` only.
- No direct provider URL is called.

## Limitations

Admin QR management remains `[B]` until backend exposes a safe admin QR contract.

Payment provider settings are `[~]` because read-only provider status is implemented, but configuration/activation/disable operations are intentionally deferred to backend-managed endpoints.
