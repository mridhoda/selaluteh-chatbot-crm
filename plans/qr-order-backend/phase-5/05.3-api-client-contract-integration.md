# 05.3 API Client and Contract Integration

Tasks: `4.1` through `4.6`

Date: 2026-07-08

Scope: Centralized Phase 5 API client boundary, safety guards, backend error mapping, Node safety tests, and contract documentation. Public UI pages were not implemented.

## Audit

Existing frontend API evidence:

- `web/src/shared/api/httpClient.js` is an axios admin/auth client that attaches token and `x-workspace-id` from storage.
- `web/src/features/public-store/api/publicStoreApi.js` is mock-only and local-first.
- `web/src/features/public-store/api/publicStoreEndpoints.js` had stale endpoint strings compared with backend route evidence.
- `web/src/shared/api/apiError.js` had only a generic message helper before Wave 1.

Backend route evidence:

- Public storefront: `GET /api/v1/public/storefronts/:storefrontSlug` and legacy alias `/stores/:storefrontSlug`.
- QR resolve: `GET /api/v1/public/qr/:qrToken`.
- Cart validate: `POST /api/v1/public/carts/validate`.
- Checkout: `POST /api/v1/public/checkout` with required `Idempotency-Key` header.
- Payment status: `GET /api/v1/public/payments/:paymentId/status`.
- Public order: `GET /api/v1/public/orders/:publicOrderToken`.
- Admin orders: `GET /api/v1/admin/orders`, `GET /api/v1/admin/orders/:orderId`, and explicit lifecycle `POST` endpoints.
- Admin payment gateway config: `GET /payments/gateway/config` is currently available through existing backend route mounting.
- Admin QR settings: no backend admin QR contract was found.

## Implementation

Files added/changed:

- Added `web/src/features/public-store/api/phase5ApiClient.js`.
- Extended `web/src/shared/api/apiError.js` with Phase 5 backend error-code mapping.
- Adjusted `web/src/shared/api/apiBase.js` so `getApiBase()` is safe when imported in Node tests outside Vite runtime.
- Added `web/test/phase-5/phase5-api-client-safety.test.mjs`.

Client structure:

- `createPhase5ApiClient(options)` creates an isolated fetch-based client.
- `phase5ApiClient` exports the default browser/runtime instance.
- `client.public.*` contains public no-login methods.
- `client.admin.*` contains Phase 5 admin methods.
- `client.request()` is exposed for tests and exceptional future shared calls, with webhook/callback guard enabled.

Base URL and headers:

- Base URL uses existing `getApiBase()` behavior from `VITE_API_BASE`, `VITE_API_URL`, or local fallback.
- Public methods do not attach admin auth/workspace headers.
- Admin methods optionally attach `Authorization` and `x-workspace-id` only when `getAuthToken` and `getWorkspaceId` callbacks are supplied.
- JSON parsing and non-2xx error normalization are centralized inside the Phase 5 client.

## Public Methods

Implemented public boundary methods:

- `getStorefront(storefrontSlug, params)` calls `GET /api/v1/public/storefronts/:storefrontSlug`.
- `resolveQr(qrToken, params)` calls `GET /api/v1/public/qr/:qrToken`.
- `validateCart(payload)` calls `POST /api/v1/public/carts/validate`.
- `checkout(payload, { idempotencyKey })` calls `POST /api/v1/public/checkout` with `Idempotency-Key`.
- `getPaymentStatus(paymentId)` calls `GET /api/v1/public/payments/:paymentId/status`.
- `getPublicOrder(publicOrderToken)` calls `GET /api/v1/public/orders/:publicOrderToken`.

Public safety rules enforced:

- Checkout throws before network if `Idempotency-Key` is missing.
- Public mutation payloads reject backend-owned fields: `payment_status`, `paymentStatus`, `fulfillment_status`, `fulfillmentStatus`, `public_order_status`, `publicOrderStatus`, `final_total`, `finalTotal`, `total_amount`, `totalAmount`, `total`, `totals`, `allowed_actions`, `allowedActions`, and `status`.
- QR token/session fields are not auto-attached globally. They are sent only when the caller supplies them to supported QR/cart/checkout payloads or path parameters.
- Payment status and public order methods are read-only.
- Webhook/provider callback paths are rejected before network.

## Admin Methods

Implemented admin boundary methods:

- `listOrders(params)` calls `GET /api/v1/admin/orders`.
- `getOrder(orderId)` calls `GET /api/v1/admin/orders/:orderId`.
- `acceptOrder(orderId, body)` calls `POST /api/v1/admin/orders/:orderId/accept`.
- `prepareOrder(orderId, body)` calls `POST /api/v1/admin/orders/:orderId/prepare`.
- `readyOrder(orderId, body)` calls `POST /api/v1/admin/orders/:orderId/ready`.
- `completeOrder(orderId, body)` calls `POST /api/v1/admin/orders/:orderId/complete`.
- `cancelOrder(orderId, body)` calls `POST /api/v1/admin/orders/:orderId/cancel`.
- `getPaymentGatewayConfig()` calls `GET /payments/gateway/config` for read-only public-safe provider config.

Admin safety rules enforced:

- No generic `PATCH /orders/:id/status` method exists in the Phase 5 client.
- Lifecycle actions use explicit `POST` endpoints only.
- Admin action payloads reject generic `status`, payment/fulfillment status authority fields, and `allowed_actions`/`allowedActions`.
- `allowed_actions` remains backend response data only; the client does not compute or guess it.
- Admin QR settings are not implemented because the backend contract is missing.

## Error Mapping

Backend error mapping was added in `PHASE5_BACKEND_ERROR_MESSAGES` and `mapBackendError()`.

Mapped codes:

- `QR_EXPIRED`
- `QR_REVOKED`
- `QR_OUTLET_MISMATCH`
- `QR_LOCATION_MISMATCH`
- `PRODUCT_UNAVAILABLE`
- `MODIFIER_INVALID`
- `CHECKOUT_IDEMPOTENCY_REQUIRED`
- `IDEMPOTENCY_KEY_REQUIRED` as runtime alias for current backend mismatch
- `IDEMPOTENCY_CONFLICT`
- `PAYMENT_PROVIDER_ERROR`
- `ORDER_INVALID_TRANSITION`
- `ORDER_UNPAID`
- `FORBIDDEN`
- `RATE_LIMITED`
- `INTERNAL_ERROR`

`getApiErrorMessage()` now delegates to `mapBackendError()` while preserving a fallback message path for older callers.

## Tests

Added RED-first safety test file:

- `web/test/phase-5/phase5-api-client-safety.test.mjs`

Test coverage:

- Checkout rejects forbidden backend-owned authority fields.
- Checkout requires and sends `Idempotency-Key`.
- Checkout payload omits final totals/payment/fulfillment status authority.
- Public methods use confirmed `/api/v1/public` endpoints.
- QR session token is only present when supplied to QR-scoped cart/checkout payloads.
- Admin lifecycle methods call explicit endpoints.
- Admin lifecycle methods never use `PATCH` or generic `/status` endpoints.
- Client rejects webhook/provider callback paths.
- Backend error mapping covers required Phase 5 codes and current idempotency alias.
