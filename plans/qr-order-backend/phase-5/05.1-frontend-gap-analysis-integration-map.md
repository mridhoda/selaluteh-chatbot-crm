Read-only endpoints:

- Storefront, QR resolve, payment status, public order tracking, admin order list/detail, payment gateway config.

Mutation endpoints:

- Cart validation is side-effect-free validation but uses POST.
- Checkout creates order/payment and requires idempotency.
- Admin order actions mutate lifecycle through explicit endpoints.

## Implementation Waves

Wave 0 complete scope:

- Test harness helpers/fixtures.
- Env guard and release gate docs.
- Existing frontend/contract audit.
- Gap analysis/integration map.

Recommended next waves:

- Wave 1: API client foundation and safety tests. Build public/admin clients, endpoint prefix config, JSON/error normalization, no webhook calls, forbidden request field tests.
- Wave 2: Public storefront and QR resolve. Integrate read-only loading/error/empty states first; avoid checkout mutations.
- Wave 3: Cart validation and checkout idempotency. Add red tests for duplicate prevention, backend totals, QR lock, and forbidden payment/fulfillment fields.
- Wave 4: Payment pending and public order tracking. Poll read-only status; public privacy tests.
- Wave 5: Admin order list/detail/actions. Use `admin-orders` explicit endpoints and backend `allowed_actions`; keep existing admin/CRM smoke checks.
- Wave 6: Admin QR/payment settings. Payment settings can extend after DTO confirmation; QR remains blocked until backend admin contract exists.
- Wave 7: Alpha hardening. Browser/device matrix, skipped/flaky policy, manual sandbox pass, regression sign-off.

Dependency order:

- API client and safety tests before UI behavior.
- Public read-only screens before checkout mutations.
- Checkout idempotency before payment pending/public order tracking.
- Admin order explicit endpoint tests before modifying existing order UI.
- No-Go checks before alpha readiness.

## File-by-File Plan

Test/harness files:

- Keep `web/test/phase-5/helpers/*.mjs` as test-only helpers.
- Keep `web/test/phase-5/fixtures/api-contract-fixtures.mjs` as mock/contract fixture layer.
- Add future API client tests under `web/test/phase-5/*api*.test.mjs` before runtime API client changes.

Public API/client files:

- Refactor or replace `web/src/features/public-store/api/publicStoreApi.js` to call backend only after tests exist.
- Reconcile `web/src/features/public-store/api/publicStoreEndpoints.js` with actual mount prefixes.
- Consider shared public API client under `web/src/shared/api` or public-store API folder to avoid admin auth header leakage.

Public UI files:

- `StorefrontPage.jsx`: load backend storefront/outlets/menu and remove mock-only assumptions.
- `CheckoutPage.jsx`: require validated cart, send idempotency header, omit backend-owned status/total fields.
- `PaymentPendingPage.jsx`: poll read-only payment status and never mark paid locally.
- `OrderStatusPage.jsx`: load by public token and render safe fields only.
- `useGuestCart.js`, `useCheckoutForm.js`, `usePaymentStatus.js`, `usePublicOrderStatus.js`, `usePublicStorefront.js`: refactor to consume API client and backend validation responses.

QR files:

- Add route/page/hook for QR token resolution only after route naming is finalized.
- Store QR token/session state only as needed; do not allow outlet/location override for locked QR.

Admin files:

- Prefer adding a Phase 5 admin order API module rather than extending empty `web/src/modules/orders/api/ordersApi.js` with legacy status patching.
- Update `OrdersPage.jsx` carefully or introduce a route-scoped component if lower regression risk.
- Add navigation only after page is tested and existing admin routes are smoke-checked.
- Keep admin QR route hidden until backend contract exists.

Environment/docs files:

- Keep `web/.env.example` public-safe.
- Do not add provider secrets or webhook secrets to frontend env.

## Regression Plan

Regression areas:

- `/app/chats` and `ChatContextPanel` cart/checkout/payment flows.
- `/app/products` product/outlet/inventory/file upload flows.
- `/app/payments` payment list/detail/provider config display.
- `/app/platforms` connected platform list/config.
- `/app/orders` existing legacy behavior until replaced.
- Login/logout/session storage behavior.

Regression method:

- Avoid modifying shared `httpClient` without tests covering admin auth headers and 401 redirect behavior.
- Introduce public API client separately if possible to avoid admin auth leakage.
- Keep frontend permissions as UI hints only and backend `allowed_actions` as action authority.
- Run manual smoke matrix after each admin route/API change.
